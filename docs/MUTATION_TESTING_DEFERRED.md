## Why deferred

### The math

Mutation testing tools generate hundreds to thousands of code mutants
per module, then run the test suite against each one. For Curator-scale
modules:

| Module | Statements | Mutants generated by mutmut (observed v1.7.150) | Wall-clock estimate per mutant | Total estimated CPU time |
|---|---|---|---|---|
| `services/migration.py` | ~1700 | **1,092** | 30 sec – 3 min | **8–50 hours** |
| `cli/main.py` | 1,843 | ~1,200 (extrapolated) | similar | **10–60 hours** |
| Each `cli/*` subcommand | 100–300 | ~80–250 | similar | 1–10 hours each |

Even a focused mutmut run with `--use-coverage` (only run tests that
exercise the mutated line) hits these orders of magnitude. The mid-range
estimate for **comprehensive mutation testing across all Curator
modules is multiple weeks of dedicated CPU time.**

That is not a ship. It is not even a single arc. It is a workstream.

### What v1.7.150 actually tried

Round 3's Tier 1 handoff (`CLAUDE_CODE_HANDOFF_ROUND3.md`) listed a
mutmut "spot check" on `services/migration.py` as a stabilization
candidate. The naming was a scope mistake by the handoff author (The
Log Claude) — Code's correct response was:

1. Started the mutmut run honestly
2. Observed the 1,092-mutants finding within minutes
3. Did the math (Lesson #88 split rule: 1.5× scope blow-up triggers a
   re-plan)
4. Killed the run
5. Reported the budget overrun candidly
6. Captured the killed run + reasoning in `docs/MUTATION_TESTING_REPORT.md`
7. Jake approved the deferral

Per partnership directive: **honest stop is better than dishonest
push-through.** This is the canonical example.

### Equivalent mutations

A non-trivial fraction of mutmut's mutants are **equivalent mutations**
— mutations that produce identical observable behavior and therefore
cannot be killed by any possible test. Example: `range(len(x))` →
`range(0, len(x))`. Same Python semantics, zero behavior difference.

Without a triage protocol set up beforehand, these flood the survivors
list and drown the real signal (actually-weak tests). Mature mutation
testing programs allocate a substantial fraction of run-time to
**equivalent-mutation triage** as a separate workstream from
test-strengthening.

## What it would take to open the arc

If mutation testing is desired in some future Round, here's what the
arc would need before opening:

### 1. Dedicated scope plan (per Lesson #88)

A document at `docs/MUTATION_TESTING_ARC_SCOPE.md`:
- Module-by-module baseline mutation count (run mutmut `--lookup` once
  per module just for the count, not the full run)
- Multi-week CPU budget estimate
- Sub-ship boundaries (one module's mutmut run per ship at minimum)
- Stop conditions (e.g. "if mutation score on services/migration.py is
  below 70% after first pass, replan")

### 2. Dedicated CPU resources

Mutation testing belongs in a **nightly CI job** or **weekend job**, not
in an interactive session. Options:
- GitHub Actions self-hosted runner (if Jake has spare hardware)
- A dedicated cloud VM allocated to mutation testing
- Local long-running job on Jake's dev box with results checked into
  the repo as artifacts

### 3. Triage protocol for survived mutations

Each survivor must be categorized:
- **Actually-weak-test:** add a test that kills the mutant → improve
  apex-accuracy
- **Equivalent-mutation:** document as such; cannot be killed; not a
  doctrine violation → list at `docs/MUTATION_SURVIVORS.md`
- **Unreachable-code:** pragma the source line per Lesson #91 →
  becomes a pragma-debt entry

A triage rubric should be written *before* the first arc opens.
Without it, the team drowns in survivor noise on first contact.

### 4. Tool selection

Re-evaluate at arc-open time:
- **mutmut 2.5.x** — what v1.7.150 attempted. Works on Windows
  (mutmut 3.x doesn't). Slow but reliable. Good editorial defaults.
- **cosmic-ray** — older, slower, larger feature set. Probably overkill.
- **mutpy** — Python 3 native, may have improved since 2020s. Worth a
  fresh look.
- New tools that may emerge — the field has moved since this doc was
  written.

## When to revisit

Signals that would justify opening the arc:

1. **A real bug ships through 100% coverage** that mutation testing
   would have caught. (Curator hasn't seen one yet, but if/when it
   happens, this is the doctrine answer.)
2. **CI capacity becomes available** — Jake invests in nightly job
   infrastructure or the equivalent.
3. **Curator approaches v3.0 / wider distribution** — mutation testing
   is appropriate quality gating for libraries used by others, less
   essential for personal-use tools.
4. **A user-facing security or correctness bug** is traced to a weak
   test — the per-bug retrospective should ask "would mutation testing
   have caught this?" If yes-and-often, the arc opens.

Until one of those signals fires, **the doctrine is: do not attempt
mutation testing as a side activity in any ship.**
