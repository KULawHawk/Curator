# v1.7.35 — `--no-autostrip` migration opt-out (closes v1.7.29 limitation)

**Date:** 2026-05-11
**Type:** User-visible feature (CLI flag) + test coverage backfill
**Scope:** `src/curator/services/migration.py` (+28 lines), `src/curator/cli/main.py` (+32 lines), `tests/unit/test_migration_autostrip.py` (NEW, 313 lines)

## TL;DR

New `--no-autostrip` flag on `curator migrate` and `curator tier --apply` lets the caller opt out of v1.7.29's auto-strip-on-public-dst behavior for an individual migration. The override is audit-logged when it actually changes behavior (dst is public). **Bonus**: this ship also adds dedicated test coverage for the v1.7.29 auto-strip path (5 new tests in a new module), which was a real coverage gap.

## Why this matters

### Closing the v1.7.29 limitation

v1.7.29 ('share_visibility auto-strip') made metadata-stripping the **default** for migrations to a `share_visibility='public'` destination source. That was the right default for privacy: a public dst means "files might be seen outside my org," and EXIF/GPS-coords/Author-fields are personally-identifying information that leak by default in most image / Office formats.

But the default had no escape hatch. To migrate to a public dst WITHOUT stripping, users had to:

1. Temporarily flip the source's `share_visibility` to `private`, OR
2. Use a separate copy mechanism that bypasses Curator's migration system entirely

Both options lose useful properties. Option 1 disables the protection for every subsequent migration; option 2 loses Curator's hash-verify-before-move + lineage-preserving + audit-logging discipline.

v1.7.35 makes the opt-out a **per-migration, audit-logged** choice that doesn't permanently change source configuration.

### Legitimate no-strip use cases

* **Forensic archival** — preserve original EXIF / docProps / PDF metadata as evidence. Stripping destroys the chain of custody.
* **Cross-system replication** — moving files to another Curator instance you control; the dst has its own privacy posture so the strip is redundant.
* **Reorganization within a public source** — moving between subfolders of the same public dst; stripping every time isn't necessary.
* **Debugging / dry-run** — verifying what would migrate without the permanent side effect of metadata removal.

## What's new

### CLI flag (two surfaces)

```bash
# Direct migration command
curator migrate local /src /dst --apply --no-autostrip

# Tier-driven migration
curator tier cold --root /src --apply --target /dst --no-autostrip
```

### `MigrationService.apply()` kwarg

```python
report = service.apply(
    plan,
    no_autostrip=True,   # NEW in v1.7.35; default False
    # ... other kwargs unchanged ...
)
```

Default `False` preserves v1.7.29 behavior. `True` skips the auto-strip phase even when the gating condition would normally trigger it.

### Audit event semantics

| Scenario | Audit event emitted |
|---|---|
| dst public + no flag (default) | `migration.autostrip.enabled` (v1.7.29) |
| dst public + `--no-autostrip` | `migration.autostrip.opted_out` **(NEW in v1.7.35)** |
| dst private/team + anything | None |

The opt-out event is logged **only when it changes behavior** (dst would have been stripped but wasn't). Quiet no-ops don't pollute the audit log:

```python
# These produce NO autostrip audit events:
service.apply(plan, no_autostrip=True)  # dst is private — opt-out is meaningless
service.apply(plan, no_autostrip=False) # dst is private — strip wasn't going to happen
# (only "dst is public + no_autostrip=True" produces the new opted_out event)
```

The `migration.autostrip.opted_out` event payload:

```json
{
  "actor": "curator.migration",
  "action": "migration.autostrip.opted_out",
  "entity_type": "source",
  "entity_id": "<dst_source_id>",
  "details": {
    "reason": "caller passed no_autostrip=True (--no-autostrip)",
    "plan_move_count": <int>,
    "dst_share_visibility": "public"
  }
}
```

## New test module: `tests/unit/test_migration_autostrip.py`

Fills a real v1.7.29 coverage gap. Before this ship, the auto-strip path had zero dedicated test coverage despite shipping in v1.7.29. The new module adds 5 focused tests using a `RecordingStripper` test double that captures `strip_file` calls without actually modifying anything:

| Test | Verifies |
|---|---|
| `test_autostrip_fires_when_dst_is_public` | v1.7.29 baseline: strip fires + `enabled` event |
| `test_no_autostrip_blocks_strip_when_dst_is_public` | v1.7.35: opt-out skips strip |
| `test_no_autostrip_audit_event_when_dst_is_public` | v1.7.35: `opted_out` event fires |
| `test_no_autostrip_is_noop_when_dst_is_private` | v1.7.35: private dst is silent |
| `test_no_autostrip_is_noop_when_no_stripper_wired` | v1.7.35: missing stripper is silent |

The `RecordingStripper` returns a synthetic `StripResult` so the migration code sees a successful outcome without actually performing any EXIF/metadata manipulation. This lets each test verify the **call signature and side effects** of auto-strip without coupling to specific format-handler behavior.

## Authoritative-principle catches

### Catch: test harness used wrong source IDs

The first test attempt used `dst_source_id="public_dst"` while seeding files under `src_source_id="local"`. This caused all 5 tests to fail with:

```
RuntimeError: cross-source: no plugin handled curator_source_read_bytes for src_source_id='local'
```

**Why**: MigrationService's `apply()` checks `src_source_id != dst_source_id` and routes to the cross-source code path (which requires plugin hooks `curator_source_read_bytes`, `curator_source_write`, and `curator_source_rename`). The unit test harness doesn't register those hooks.

**Fix**: keep `src_source_id == dst_source_id == "local"` and update the local source's `share_visibility` attribute in place. The auto-strip gating condition checks `dst_source.share_visibility == "public"` regardless of whether src == dst, so this stays on the same-source local-FS code path while still triggering auto-strip.

**Codified as lesson #55**: when testing a feature whose gating depends on a source ATTRIBUTE (not source IDENTITY), modify the existing source in place. Don't register a new source if you can avoid it — that changes the code path.

### Lesson #55 (new this ship)

> **When writing tests for a feature whose gating depends on a *source attribute* (e.g. `share_visibility`, `enabled`, plugin type) rather than on *source identity*, modify the existing source's attribute in place instead of registering a new source. Cross-source code paths in MigrationService and similar services require plugin hooks (`curator_source_read_bytes`, `curator_source_write`, `curator_source_rename`) that unit-test harnesses typically don't provide; staying same-source keeps the test on the local-FS code path. If the feature legitimately requires cross-source semantics, write the test under `tests/unit/test_migration_cross_source.py` where the plugin-stub harness is already established.**

The cost of this bug: ~5 minutes (write tests → see all 5 fail with the same error → realize cross-source path was triggered → switch to same-source with attribute update → all 5 pass). The mistake was a category error rather than a logic bug. Easy to make, easy to fix once diagnosed.

## Limitations

- **Phase 1 only.** `MigrationService.apply()` got the kwarg; `MigrationService.run_job()` (Phase 2 persistent path) did NOT. v1.7.29 only wired auto-strip into Phase 1 — the persistent path was already a `--no-autostrip` no-op. If Phase 2 ever gains auto-strip, the flag must be plumbed to `run_job()` too. Documented in the service-level docstring.

- **No per-file granularity.** The flag is per-migration. To strip some files but not others, run two separate migrations (or subclass `MigrationService`). The simpler design fits the common case; per-file control isn't a use case anyone has reported.

- **No `--strip` (force-enable) flag.** Only the opt-OUT direction is added. There's no force-enable on private destinations because the user can just call `curator export-clean` directly if they want to strip files they control. The asymmetry is deliberate.

- **GUI parity not shipped.** The CLI gets the flag; the GUI's migration dialog (v1.7.27) does not yet have an equivalent toggle. v1.7.27's TierDialog can drive a CLI-via-subprocess migration that accepts the flag, but the in-process GUI path uses the service API directly without exposing this kwarg. Future GUI ship can address.

- **Audit event detail count is hardcoded.** The `migration.autostrip.opted_out` event records `plan_move_count` from the plan, not the actual count of files that would have been stripped (which depends on per-file outcomes). For most cases these are equivalent; for partial-failure scenarios they may diverge. Future ship could enrich the details if needed.
