## Lessons captured

**Lesson #82 — Orchestrator modules CAN reach 100%; the question is stub infrastructure, not feasibility.**
My initial assessment was pessimistic. The reality: 3 passes, ~620 lines of test code (8 stubs + 23 tests), 100%. The 100% bar is almost always achievable if you're willing to build the fakes.

**Lesson #83 — Monkeypatching service METHODS is safer than monkeypatching stdlib classes.**
`monkeypatch.setattr(svc, "_stat_to_file_info", boom_stat)` works cleanly. `monkeypatch.setattr(Path, "stat", boom_stat)` breaks `Path.exists()` as collateral. Prefer the helper boundary.

**Lesson #84 — Stub-first architecture pays compound interest.**
Each new Phase Gamma module's stubs follow the same design vocabulary. What took longer in v1.7.82 (lineage) felt mechanical by v1.7.86 (scan). Transferable skill, not per-module artifact.

**Lesson #85 — Push back on your own recommendations when evidence changes.**
The trash test hang flag from v1.7.84/85 was retracted this session after evidence showed the suite runs clean. The partnership directive cuts both ways: push back on Jake's plans when they look wasteful, AND push back on my own prior plans when evidence makes them moot.

**Lesson #86 — "Mid-ship" is not an acceptable session-end state. Don't leave defective processes.**
I hit a context-window limit mid-way through this ship and presented "shipped at 100% on disk, ceremony pending" as a reasonable stopping point. Jake pushed back hard: *"it doesn't matter the time commit get it untangled and 100%. why leave a defective process."* He's right. The repo on disk and the repo on origin/main are not the same state. Forked state means:

1. The next session's context summary will report "v1.7.86 mid-ship" — a forked-state signal that costs more time to interpret than just finishing would have.
2. CI doesn't run against on-disk code; it runs against origin/main. An untagged ship has zero CI verification.
3. The git history loses the link between the work and the narrative. Recovering that link later is more expensive than writing it now.
4. "Done at 100%, almost shipped" reads to a future me as "essentially done." **A ship is committed, tagged, and pushed, or it isn't a ship.**

This is the inverse of Lesson #71 (apex accuracy): just as "94% coverage" is corner-cutting language for accuracy, "mid-ship" is corner-cutting language for the ceremony. **When a ship's work is done at 100%, the next action is ALWAYS commit/tag/push, period.** Token-budget pressure is not an exception — it's the moment when the discipline matters most.
