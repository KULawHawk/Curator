## What's new

### TTY + encoding detection
```python
_enc = (_sys.stdout.encoding or "").lower().replace("-", "")
_bar_ch = (
    "\u2588"
    if _sys.stdout.isatty() and _enc.startswith("utf")
    else "#"
)
```

Both conditions must be true; otherwise fall back to `#`. Catches:
- Subprocess pipes (`isatty=False`) ‚Üí `#`
- Windows `cmd.exe` with `cp1252` ‚Üí `#`
- File redirects ‚Üí `#`
- Modern Windows Terminal / VS Code / macOS / Linux TTY ‚Üí `‚ñà`

### Encoding normalization
Strips dashes and lowercases so `utf-8`, `UTF-8`, `utf8`, `UTF_8` all match.

### Conservative defaults
Missing encoding (`None` or `""`) ‚Üí `#`.

### No flag added
Auto-detection is transparent. No opt-in required.

## Authoritative-principle this turn

**0 bugs caught** ‚Äî clean first-try ship.

**Lesson #52 logged**: when shipping a feature whose behavior depends on runtime context (TTY mode, terminal capabilities, env vars), test the **selection logic** as a pure function and the **fallback path** via subprocess. The non-fallback path is verified by correct selection logic + manual smoke test in a real terminal.

Subprocess output is always non-TTY by definition, so the test design replicates the selection function in the test file and exercises it directly with all 9 input combinations. The subprocess regression tests confirm the fallback path works correctly.

## Limitations

- **No `--ascii-bars` / `--unicode-bars` override flags** ‚Äî could force one or the other regardless of detection
- **Trust the `encoding` field** ‚Äî if `PYTHONIOENCODING` is set weirdly, behavior is undefined
- **Manual smoke test required for TTY path** ‚Äî automated tests can only verify the non-TTY pipe case
- **No shared helper** ‚Äî if more commands grow histograms, the logic should be extracted to `cli/util.py:bar_char()`
