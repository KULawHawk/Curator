# v1.7.62 — Strip ANSI + use result.output in 3 Rich-help tests

**Date:** 2026-05-12
**Type:** Test-level fix for CI cross-platform compatibility
**Scope:** 3 test files (one assertion site each)

## TL;DR

v1.7.61's `COLUMNS=200` env var didn't fix the 3 failing tests (CI re-ran with same `1808 passed, 3 failed`). PAT-enabled log analysis showed the real cause: Rich's help output goes to a stream `CliRunner.result.stdout` doesn't capture on POSIX. **Fix: use `result.output` (combined stdout+stderr) + strip ANSI codes via regex before substring matching.**

## How we got here

After v1.7.61 shipped, CI re-ran and produced identical failures on the same 3 tests:

```
FAILED test_cli_cleanup_duplicates.py::test_duplicates_help_lists_strategies
    AssertionError: assert '--keep-under' in result.stdout
FAILED test_cli_migrate.py::test_migrate_help_shows_phase_1_note
    AssertionError: assert '--apply' in result.stdout
FAILED test_organize_mb_enrichment.py::test_help_lists_enrich_mb
    AssertionError: assert '--enrich-mb' in result.stdout

3 failed, 1808 passed, 6 skipped, 10 deselected in 156.68s
```

Grep of the entire 115KB CI log: substring `apply` literally never appears in any captured stdout content — only in the AssertionError messages echoing the failed asserts. The option names are being rendered to a stream `result.stdout` doesn't see.

## Root cause

Click's `CliRunner`:
- `result.stdout` — only stdout (when `mix_stderr=False`)
- `result.output` — combined stdout + stderr (always)

On Windows, Rich's Console happens to write the help table to stdout. On POSIX, Rich's Console with default settings writes to stderr (or a separate file descriptor that bypasses click's stdout buffer). The tests asserted against `result.stdout`, which on POSIX contains only the table frame, not the option list.

## The fix

```python
import re
output = re.sub(r"\x1b\[[0-9;]*m", "", result.output)
assert "--apply" in output
```

Two changes per assertion site:
1. **`result.stdout` → `result.output`**: captures whatever stream Rich uses
2. **Strip ANSI codes**: defensive against future Rich versions; makes the test invariant to terminal-width and color settings

## Catches

- **v1.7.61 was a speculative fix.** I hypothesized COLUMNS=200 would solve a wrap; it didn't. The PAT-enabled diagnosis revealed the actual issue (stream routing) and the correct fix (3-file test edit) in <10 minutes.
- **The robust fix was already documented as future work in v1.7.61's CHANGELOG.** "A more robust fix would strip ANSI codes before substring matching..." — the minimal infra fix turned out to be the wrong gamble. Should have done the 3-file test edit from the start.
- **`result.output` over `result.stdout` is the cross-platform default** for click CLI tests, especially those exercising Rich rendering. Subordinate canonical pattern.
- **In-line `import re` accepted** for diff minimality. Future refactor can hoist if useful.

## Lesson #67 codified

**"Diagnose with logs, not with hypotheses."** v1.7.61's COLUMNS=200 fix was a reasonable inference from the symptom (ANSI codes + spaces + truncation in assertion message). It was wrong. The PAT-enabled diagnostic loop should be the FIRST move when CI is red, not the third. Mitigations:
- Persist the PAT somewhere (with clear scope notice) so it's not re-pasted each time
- Add a `scripts/ci_diag.ps1` helper that downloads the latest failed run's logs in one command
- Resist shipping speculative fixes when log access is available

Reinforces lesson #66 ("green local pytest does not imply green CI") — fifth ship in this sub-arc.
