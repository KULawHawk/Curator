# v1.7.71 — Dependabot config: automated GitHub Actions version tracking

**Date:** 2026-05-12
**Type:** CI automation (dependency tracking)
**Scope:** `.github/dependabot.yml`

## TL;DR

v1.7.67 manually bumped 3 actions to Node.js 24 versions — a process that required research, web searches, and 3 weeks of buffer before GitHub's forcing date. **This ship configures Dependabot to watch GitHub Actions versions automatically**, opening a grouped weekly PR whenever any action publishes a new version. Future deprecation cycles get detected within days, not 8 months.

## Why this matters

Node.js 20 deprecation timeline:
- **2025-09-19**: deprecation announced
- **2025-08 to 2025-12**: v5/v6 of all 3 actions published with Node 24 support
- **2026-05-12 (v1.7.67)**: manual bump shipped — **8 months after deprecation**
- **2026-06-02**: GitHub forces Node 24 (would have been a fire-drill)

Dependabot would have surfaced these bumps within ~1 week of each action's v5 release. Manual tracking failed for 8 months. This ship eliminates that failure mode for future deprecation cycles.

## Config design

```yaml
version: 2
updates:
  - package-ecosystem: github-actions
    directory: /
    schedule:
      interval: weekly
      day: monday
      time: "08:00"
      timezone: America/Chicago
    groups:
      github-actions-all:
        patterns: ["*"]
    open-pull-requests-limit: 5
    commit-message:
      prefix: "deps"
      include: scope
    labels: [dependencies, github-actions]
```

Key design choices:
- **Weekly Monday 08:00 America/Chicago** — matches project timezone
- **Group all actions into one PR** — mirrors v1.7.67's successful test pattern (3 bumps in one ship, validated by 9-cell matrix)
- **`open-pull-requests-limit: 5`** — prevents queue buildup during review delays
- **`deps(github-actions): ...` prefix** — conventional-commit-style for `git log --grep "^deps"` filtering
- **Labels** for triage in GitHub UI

## Why NOT pip ecosystem (yet)

Python deps are deliberately not auto-tracked:
- Plugin ecosystem deps are version-pinned by design (send2trash backends, PyDrive2, protobuf)
- Compatibility surface is large (Qt6, pluggy, loguru, SQLite, MusicBrainz)
- CI signal must stabilize first — the v1.7.42-v1.7.64 red arc just closed

Future ship: add `pip` ecosystem when CI has been 9/9 GREEN for 25+ consecutive ships.

## Catches

- **Weekly, not daily** — actions release every 1-3 months; daily would create noise
- **Grouped, not separate PRs** — mirrors v1.7.67's successful 3-in-1 test cycle
- **Explicit timezone** — implicit UTC could land PRs at inconvenient times
- **Pip deferred, not skipped** — documented reasoning in config header
- **Cap at 5 open PRs** — survives review delays without piling up
- **No auto-merge** — CI gate + human review (intentional)

## What this does NOT do

- Doesn't auto-merge dependabot PRs
- Doesn't watch Python deps
- Doesn't watch Docker images (no Dockerfile)
- Doesn't separate security-only updates
- Doesn't backfill PRs for the bumps already done in v1.7.67

## Lessons reinforced

- **Automated trackers prevent manual-tracking gaps.** 8 months → 1 week.
- **Group automated changes by test cycle.** Mirrors how humans validate (run the matrix, see all bumps validated together).
