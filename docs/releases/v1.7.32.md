# v1.7.32 — Lesson #50 pytest-level lint (regression guard)

**Date:** 2026-05-11
**Type:** Test-level structural defense
**Scope:** `tests/unit/test_cli_util.py` (+132 lines, no production code changed)

## TL;DR

A single new pytest test scans `src/curator/cli/` for the 8 dangerous Unicode codepoints from `_GLYPH_FALLBACKS` and fails with a self-explanatory error message if any are found outside `cli/util.py`. Closes lesson #50 at the test level — future regressions are caught the next time anyone runs the test suite, with the failure message telling the developer exactly what to do.

## What the new test does

`test_no_literal_glyphs_in_cli_outside_util` is a pytest function that:

1. Discovers `src/curator/cli/` relative to the test file (no hardcoded paths)
2. Walks every `.py` file in that directory tree, skipping `util.py` itself
3. Examines every line for the 8 codepoints in `_GLYPH_FALLBACKS`: `U+2588` `U+2713` `U+2717` `U+2192` `U+2190` `U+2026` `U+00D7` `U+26A0`
4. Skips comment-only lines (line strips to start with `#`)
5. If any glyphs found, calls `pytest.fail()` with a structured message containing:
   - File, line number, glyph name, codepoint
   - The offending source line (truncated to 80 chars)
   - The canonical import statement to use as the fix
   - A one-paragraph explanation of WHY the fix is needed

## Example failure (what a future regression looks like)

If a contributor wrote `console.print(f"[green]\u2713[/] done")` somewhere in `cli/main.py`, the next `pytest` run would produce:

```
FAILED tests/unit/test_cli_util.py::test_no_literal_glyphs_in_cli_outside_util

Found 1 literal Unicode glyph(s) in src/curator/cli/ outside util.py:

  cli/main.py:L1234  CHECK (U+2713)  in: 'console.print(f"[green]\u2713[/] done")'

Fix: import constants from curator.cli.util instead of using literal glyphs:
  from curator.cli.util import (
      CHECK, CROSS, ARROW, LARROW, ELLIPSIS,
      BLOCK, TIMES, WARN, safe_glyphs,
  )

Why: literal glyphs crash the cp1252 encoder when stdout is captured
by a subprocess test or piped to a file. The constants fall back to
ASCII automatically. See v1.7.30 release notes for the 5-strike history.
```

Everything the developer needs to fix the issue is in the failure message itself. No archaeology, no Stack Overflow, no waiting for code review.

## Scope justification

The lint only checks `src/curator/cli/` because that's where the cp1252 crash risk materializes. CLI commands print to stdout; subprocess tests capture that as cp1252-encoded bytes. Other directories may legitimately contain the same glyphs:

| Directory | Legitimate glyph usage | Why safe |
|---|---|---|
| `gui/dialogs.py`, `gui/models.py` | Qt button labels, table headers | QString is UTF-16 internally; never flows to cp1252 encoder |
| `services/`, `models/`, `storage/` | Module docstrings, comments | Not executed; not printed to stdout |
| `_vendored/` | Vendored third-party docstrings | Do not modify; not our code |

The scope choice is documented in the test docstring so a future maintainer can widen the lint (e.g., to add a Qt-aware exclusion for `gui/`) or narrow it (if a false positive shows up) with full context.

## Design lessons reinforced

1. **Codify defenses at the most automated level available.** Each step in the progression made the defense more automated: ad-hoc fixes → helper module → test lint. The next theoretical step would be a pre-commit hook, then a CI required check. Each layer is independent and additive.

2. **Pytest test beats separate CI script.** A separate `scripts/check_no_raw_glyphs.py` invoked from `pre-commit.com` or a GitHub Action would be more decoupled but requires setup, documentation, and developer discipline to install. A pytest test runs automatically with every `pytest tests/` invocation — the same command developers already run before commit. Zero setup, zero maintenance, sub-second runtime.

3. **Failure messages should include the fix.** The `pytest.fail()` message tells the developer not just "there's a glyph at L1234" but "here's the exact import statement and the reason." When a regression happens six months from now, the developer doesn't need to find this CHANGELOG entry to know what to do.

4. **Scope the lint to where the failure actually happens.** A naive lint scanning the whole `src/` tree would flag docstrings, Qt strings, and vendored code as false positives. Scoping to `src/curator/cli/` matches the actual cp1252 risk surface. False positives kill lint adoption faster than false negatives.

## Limitations

- **Comment-only line detection is string-based, not AST-based.** A line like `code()  # contains \u2713` is technically a code line with a glyph in a comment; the test flags it. In practice the codebase has no such mixed lines; the simpler check is fine. `ast.parse()`-based detection would be more correct but more complex.

- **`gui/` is not checked.** Qt widget text is safe today (UTF-16 internally). If any GUI text ever flows to a CLI capture path, that's a regression the test won't catch. Future widening: scope to `gui/` too, with exclusions for `setText()`, `setWindowTitle()`, `setHtml()`, `setToolTip()` Qt API calls (which reach Qt's UTF-16 layer, not stdout).

- **Only 8 codepoints checked.** If a new glyph type is introduced (e.g., emoji `🎉`, math `∑`, CJK), it won't be caught until someone hits the cp1252 crash. Cure: adding the new entry to `_GLYPH_FALLBACKS` in `cli/util.py` automatically picks it up in the test on the next run.

- **No pre-commit integration.** Developers who push without running pytest can still introduce a glyph. A future `.git/hooks/pre-commit` or `pre-commit.com` config would close that gap. Not urgent because most developers run pytest as part of normal development, and the test runs in ~600ms.

## Lesson #50 final status

After 6 ships of effort, the defense is now three-layered:

```
                                ┌──────────────────┐
                                │  CONTRIBUTOR     │
                                │  writes new CLI  │
                                │  print statement │
                                └─────────┬────────┘
                                          │
              ┌───────────────────────────┴───────────────────────────┐
              │                                                       │
        Uses helper                                            Uses literal glyph
        (the easy path)                                        (the unsafe path)
              │                                                       │
              ▼                                                       ▼
   ┌──────────────────────┐                                ┌──────────────────────┐
   │ cli/util.CHECK       │                                │ Pytest lint fails on │
   │ falls back to "[OK]" │                                │ next test run with   │
   │ under non-TTY        │                                │ helpful fix message  │
   └──────────────────────┘                                └──────────────────────┘
              │                                                       │
              ▼                                                       ▼
        ✅ Safe code                                          ❌ Test failure with fix
        ✅ No crash                                           ⚙️ Developer fixes
```

Classified: **closed**. Future strikes would require active subversion of all three layers — the helper, the test, AND the CHANGELOG history.
