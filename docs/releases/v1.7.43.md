# v1.7.43 — Add `pytest-timeout` to `[dev]` extras (first CI-caught bug)

**Date:** 2026-05-11
**Type:** Dependency fix (1-line pyproject change)
**Scope:** `pyproject.toml` (+1 line)

## TL;DR

CI shipped 20 minutes ago. Its first run on `81a84a1` failed within 4 minutes with `pytest: error: unrecognized arguments: --timeout=120`. The local `.venv` had `pytest-timeout` installed manually; pyproject didn't declare it. Any new contributor running `setup_dev_env.py` would have hit the same error. Fix: add `pytest-timeout>=2.0` to `[dev]` extras.

## Diagnosis

```
PS> pip show pytest-timeout
Name: pytest-timeout
Version: 2.4.0
```

Locally installed. Not in pyproject. The `--timeout` flag has been used by:

- `scripts/run_pytest_detached.ps1` (v1.7.39, `--timeout=20`)
- `.github/workflows/test.yml` (v1.7.42, `--timeout=120`)
- All my recent ad-hoc pytest invocations

All of these silently depend on `pytest-timeout` being installed in the test env. Locally I had it; CI didn't.

## Fix

`pyproject.toml`:

```toml
dev = [
    "pytest>=8.0",
    "pytest-asyncio>=0.23",
    "pytest-qt>=4.2",
    "pytest-timeout>=2.0",   # NEW v1.7.43
    "hypothesis>=6.95",
    "ruff>=0.3",
    "mypy>=1.8",
]
```

The `>=2.0` floor is the version that introduced the `--timeout` CLI flag we depend on; my local has 2.4.0.

## Authoritative-principle catches

### Catch — undeclared local deps are structurally invisible to local-only dev

The host environment carries assumed-installed packages that aren't declared in the project's manifest. Code uses those features → works locally → breaks on every fresh install. **The only systematic way to catch this is a clean-environment CI run.** Manual care isn't enough — humans don't audit which packages are "in the project" vs "installed for one-off reasons years ago" unless forced to.

No new lesson codified — this IS lesson #61 in concrete form.

## Limitations

- **CI run #1 stays as failed in history.** GitHub retains it; not editable. Run #2 will be the first green run.
- **Other latent missing deps may exist.** This ship fixes the specific one CI flagged. If the test suite uses other plugins not in `[dev]`, they'll surface on next CI run. Pattern: add to `[dev]`, reship.
- **The local `.venv` had `pytest-timeout` from an unknown manual install.** Future contributors recreating their venv via `setup_dev_env.py` get the correct deps automatically.
