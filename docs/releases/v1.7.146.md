## Investigation summary

Per the handoff's three hypotheses:

1. **Different test selections produce different coverage** ✅ confirmed.

| Invocation | Result |
|---|---|
| `pytest tests/unit/` | 99.71% |
| `pytest tests/ -k "migrat"` | 100.00% |
| `pytest tests/` | 100.00% |

The "100%" at v1.7.107 used `-k "migrat"` (CLAUDE.md's canonical migration-focused selection). Unit-only fell to 99.71% on lines 505→509 (plan dst_source_id explicit-arg) and 984-988 (`_execute_one` cross-source dispatcher arm).

2. **Integration vs unit coverage divergence** ✅ root cause. Integration tests exercise the full `apply() → _execute_one → dispatch` chain; unit tests use direct-call `_execute_one_cross_source(...)` and bypass the dispatcher (Lesson #90 pattern).

3. **Coverage-tool measurement edge case** ✗ not the cause.

## Lesson captured

No new lesson — Lesson #93 (re-measure baselines) caught this in the very first ship of Round 3, validating its value even on modules previously marked "done".

**Doctrine clarification (carried into Round 3 procedure, not a new numbered lesson):** Canonical coverage measurement should use the full suite (`pytest tests/`) or the focused arc selection (`-k "..."`); the unit-only invocation is the fast feedback loop, not the canonical measure. Modules with orchestration code (especially the larger services) should have BOTH unit-direct AND integration-dispatch coverage so unit-only measurement remains a useful proxy.
