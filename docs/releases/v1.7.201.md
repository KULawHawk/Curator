## 🐛 Bug — `_check_mcp_probe` contract violation

`HealthCheckDialog`'s class docstring says: *"The dialog never raises — every check is wrapped in a try/except that converts unexpected errors into a `_CheckResult(passed=False)`."* But `_check_mcp_probe` at line 1031 accessed `self.runtime.config.db_path` **outside** its try/except. A `runtime.config` raise would propagate through `refresh()` and crash dialog construction.

**Discovery:** `TestFilesystemCheck::test_filesystem_exception` used Lesson #104 / Doctrine #22 — installing a `_RaisingConfig` real class with `@property` that raises — to exercise the filesystem-check defensive boundary. Instead of the filesystem check capturing the exception, the test failed with `RuntimeError("config fail")` propagating out of `_check_mcp_probe`.

**Fix:** moved the env-setup (`Path(self.runtime.config.db_path).parent / "curator.toml"`) inside the existing try block. Added a 5-line documenting comment.

**Severity:** medium. Would have manifested in production only if `runtime.config` raised mid-session (e.g. config file deleted while Curator was running). Unlikely but real contract violation. Was undetected since v0.34 (when the dialog was added).
