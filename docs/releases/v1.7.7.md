# Curator v1.7.7 — Release Notes

**Tagged:** 2026-05-11
**Tag:** `v1.7.7`
**Theme:** T-B07 Metadata-Stripping Export Pipelines — privacy-safe sharing.

---

## Headline

New `curator export-clean <src> <dst>` command copies files while stripping privacy-leaking metadata. Handles **images (EXIF/XMP/IPTC/PNG-text), DOCX (author/company props), and PDF (metadata dict)** in one pass. Source files never modified.

---

## What's new

### Service layer: `services/metadata_stripper.py` (~330 lines)

```python
from curator.services.metadata_stripper import MetadataStripper

stripper = MetadataStripper()  # keep_icc_profile=True, jpeg_quality=95

# Single file
result = stripper.strip_file("photo.jpg", "photo_clean.jpg")
print(result.outcome)                    # StripOutcome.STRIPPED
print(result.metadata_fields_removed)    # ['exif']

# Directory walk
report = stripper.strip_directory(
    "./client_files", "./client_files_shareable",
    recursive=True, extensions=[".jpg", ".docx", ".pdf"],
)
print(report.stripped_count, report.failed_count)
```

### 4 format handlers

| Format | Extensions | What's stripped | What's kept |
|---|---|---|---|
| **Image** | `.jpg .jpeg .png .tiff .tif .webp` | EXIF, XMP, IPTC, PNG text chunks, JPEG comments | Pixel data, ICC profile (default; toggle via `--drop-icc`) |
| **DOCX** | `.docx .docm .dotx .dotm` | docProps/core.xml + app.xml (author/company/dates), docProps/custom.xml | All content (word/document.xml, styles, media) |
| **PDF** | `.pdf` | /Author /Creator /Producer /Title /Subject /Keywords /CreationDate /ModDate | Pages, content streams |
| **Other** | * | nothing | byte-for-byte passthrough |

### CLI: `curator export-clean <src> <dst>`

```
$ curator export-clean ./photos ./photos_clean --show-files

Metadata export-clean
  Source:           ./photos
  Destination:      ./photos_clean
  Duration:         0.42s
  Total files:      24
  Stripped:         24

Per-file outcomes:
     stripped  photos/vacation.jpg (exif)
     stripped  photos/portrait.png (png_text_chunks)
     ...
```

Flags:
- `--recursive / --no-recursive` (default: recursive)
- `--ext .EXT` (repeatable filter)
- `--drop-icc` (also strip ICC color profiles from images)
- `--show-files` (per-file outcomes; default = summary + failures only)
- `--json` (machine-readable output)

## Authoritative-principle catches

**1 Pillow API bug caught on first test run:**
- `Image.save(..., quality='keep')` requires the source image to still have its JPEG quantization tables attached. `im.copy()` strips them (the copy is just pixel data + mode). Fixed by defaulting to numeric `jpeg_quality=95` — slightly lossy on re-encode but predictable. Lesson logged: **Pillow's `quality='keep'` mode and `im.copy()` are incompatible**; the docs don't make this obvious.

**Append-pattern lesson applied from v1.7.6:**
- Last release's `Filesystem:edit_file` append diff silently de-indented adjacent untouched code. This release used a **PowerShell read-content + string-substitute** pattern instead: load main.py as a single string, find the unique `if __name__ == '__main__':` marker, insert before it, write back, then `ast.parse` to verify. Zero collateral damage.

## Limitations / next steps

- **No per-source policy** — you have to run `curator export-clean` explicitly. A v1.8 `SourceConfig.strip_metadata` / `share_visibility` field will auto-gate migration destinations.
- **No integration with `curator migrate` or `curator organize --stage`** — drop-in candidate for a v1.8 hookspec.
- **No XMP-in-PDF body stripping** — some PDFs embed XMP packets in the file body (not just the /Metadata dict). pypdf doesn't strip those. A `qpdf` shell-out would catch them but adds a system dep.
- **JPEG re-encode is lossy** — `quality=95` incurs minor quality loss vs the original. Acceptable for a privacy-export pipeline (sharing is the goal, not archival fidelity).
- **DOCX orphan references** — we don't currently update `_rels/.rels` to drop references to the deleted `docProps/custom.xml`. Most readers tolerate this but strict validators could complain.
- **No GUI integration** — CLI-only for now.
