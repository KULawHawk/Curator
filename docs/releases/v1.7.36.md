### Windows blank-line fix

Discovered while smoke-testing `audit --csv`: each row had a visible blank line between it and the next. Root cause:

```python
# Before (broken on Windows console):
writer = _csv.writer(sys.stdout)
# csv.writer default lineterminator='\r\n'
# Python text-mode stdout newline translation: '\r\n' -> '\r\r\n' (DOUBLES the \r)
# Net result on Windows console: extra blank line between every row

# After (clean on every platform):
writer = _csv.writer(sys.stdout, lineterminator="\n")
# Single '\n' terminator; text-mode stdout translates to platform-appropriate
# newline. No doubling. Clean output everywhere.
```

Applied to all 7 stdout-writing csv.writer sites (3 from v1.7.33 + 4 new in v1.7.36). The file-writing CSV in `audit-export` was already correct (writes to a file handle opened with `newline=''`) and unchanged.

**Verification** by `repr(stdout)`:

```python
# Before fix:
'audit_id,...\r\r\n148,...\r\r\n147,...\r\r\n'

# After fix:
'audit_id,...\n148,...\n147,...\n'
```

This bug had been latent in v1.7.33's three CSV commands since they shipped. v1.7.33's smoke test caught the structure but not the blank lines because `csv.reader` is tolerant of blank lines between records — it hides the bug. v1.7.36's test suite includes a `_expect_no_blank_lines` helper that asserts `'\n\n' not in stdout`, which would have caught it.

## Authoritative-principle catches

### Catch: Windows blank-line in CSV (latent since v1.7.33)

First smoke test of `audit --csv` showed:

```
audit_id,occurred_at,actor,action,entity_type,entity_id,details
                                                                ← BLANK LINE
148,2026-05-11T17:36:16.847,cli.audit,audit.exported,...
                                                                ← BLANK LINE
147,...
```

Investigation:
1. Hypothesis: csv.writer adding extra newline
2. Test: `repr(stdout)` showed `\r\r\n` per row
3. Diagnosis: csv.writer default `\r\n` + text-mode stdout newline translation on Windows = `\r\r\n`
4. Fix: `lineterminator='\n'` on every stdout csv.writer
5. Verify: re-run shows clean `\n` per row

End-to-end fix took ~10 minutes once the diagnostic was clear. The same bug had been in v1.7.33's three CSV commands since they shipped (one month? two weeks?) without anyone noticing because:
- `csv.reader` (the parsing side) is tolerant of blank lines
- Most users redirect CSV to files where the blank lines may not be visually noticed
- The bug is Windows-console-specific

### Lesson #56 (new this ship)

> **When emitting CSV to stdout on cross-platform code, always pass `lineterminator='\n'` to `csv.writer()`. Python's text-mode stdout adds platform-appropriate newline translation; csv.writer's default `\r\n` then doubles up to `\r\n\r\n` on Windows, producing visible blank lines between rows on the console. The fix preserves CSV correctness (any RFC 4180-compliant tool accepts `\n`-terminated lines) and eliminates platform-specific visual noise. Always verify CSV output via `repr(stdout)`, not just `csv.reader(stdout)` parsing — csv.reader is tolerant of blank lines, so it hides the bug.**

This generalizes to any line-oriented output to stdout: verify line terminators by `repr()` before trusting tool-side parsers.

## Limitations

- **`inspect` and `gdrive paths` don't get CSV.** Single-record information dumps; CSV would be silly.
- **`cleanup *` commands don't get CSV.** Findings + apply reports have complex nested structure; CSV would lose information.
- **`safety check` / `safety paths` don't get CSV.** Structured analysis, not records.
- **`details` / `config` columns are JSON-encoded inline.** Full information preserved; downstream tools wanting per-key columns use `--json` instead.
- **No TSV dialect yet.** v1.7.33 limitation 1 remains open. ~20 lines of follow-up.
- **No CSV-dialect override flag.** All 10 commands use RFC 4180 (QUOTE_MINIMAL). Users wanting different quoting (e.g., QUOTE_ALL for tools that hate unquoted dates) must post-process.
