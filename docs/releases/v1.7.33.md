The pattern is now complete for analytics-shaped commands. Users running cross-system pipelines (Grafana dashboards, weekly capacity-planning Excel sheets, ML training data extraction) get a stable record-by-row output without the post-processing tax.

Example flows that work after this ship:

```bash
# Pipe forecast into awk to alert on imminent fill-up
curator forecast --csv --no-header \
    | awk -F, '$5 > 80 {print $1": "$5"% full"}'

# Bulk-strip a directory and feed results into a SQLite analysis tool
curator export-clean ./client_files ./public --csv > stripped.csv
sqlite3 audit.db ".import --csv stripped.csv stripped"

# Pipeline cold-tier candidates into a duckdb query for size-by-source
curator tier cold --csv --limit 1000 \
    | duckdb -c "SELECT source_id, SUM(size)/1e9 AS gb \
                FROM read_csv_auto('/dev/stdin') GROUP BY 1"
```

### The lesson #50 strike #6 (the more important story)

v1.7.32 codified the lint at the test level. v1.7.33 was its **first live save**:

1. While reading `forecast_cmd`'s body for the JSON-output pattern (to copy the structure for CSV), spotted a literal `R²` in the slope-rate `console.print(...)` call
2. Hypothesis: latent cp1252 crash waiting for the right subprocess test capture
3. Added `"\u00b2": "^2"` to `_GLYPH_FALLBACKS` and `SUPER2` constant
4. Updated `DANGEROUS_GLYPHS` in the lint test to include `\u00b2`
5. Ran `pytest tests/unit/test_cli_util.py`. **Test failed** with a precise pointer:
   ```
   FAILED test_no_literal_glyphs_in_cli_outside_util
   Found 1 literal Unicode glyph(s) in src/curator/cli/ outside util.py:
     cli/main.py:L3449  SUPER2 (U+00B2)  in: f"  (R²={f.fit_r_squared:.3f})"
   Fix: import constants from curator.cli.util instead of using literal glyphs:
     from curator.cli.util import (CHECK, CROSS, ..., safe_glyphs)
   ```
6. Fix: `f"  (R{SUPER2}={f.fit_r_squared:.3f})"`
7. Re-ran tests. **Pass.**

End-to-end loop time: <60 seconds. No production crash, no end-user report, no archaeology. The v1.7.32 release-notes claim that "adding a new codepoint to the table automatically picks it up in the test on the next run" was proven in this ship.

Empty cells for `slope_gb_per_day` / `fit_r_squared` when the drive has insufficient history (<2 months). Empty cells for `days_to_*pct` / `eta_*pct` when the drive is already past threshold or has no growth.

### `export-clean` CSV (one row per file result)

```bash
$ curator export-clean ./photos ./clean --csv | head -3
source,destination,outcome,bytes_in,bytes_out,metadata_fields_removed,error
C:\photos\IMG_0042.jpg,C:\clean\IMG_0042.jpg,stripped,2845192,2840441,exif|xmp|iptc,
C:\photos\thumbnail.gif,C:\clean\thumbnail.gif,passthrough,15022,15022,,

Columns: `source`, `destination`, `outcome` (`stripped` / `passthrough` / `skipped` / `failed`), `bytes_in`, `bytes_out`, `metadata_fields_removed` (pipe-delimited if multiple), `error` (empty unless `outcome == failed`).

Honors the exit-code-1 contract: if any file failed, exits 1 after emitting CSV (matches Rich and JSON modes).

### `tier` CSV (one row per candidate)

```

Columns: `curator_id`, `source_id`, `source_path`, `size`, `status`, `last_scanned_at`, `expires_at`, `reason`.

Honors `--limit` (CSV output is limited the same way Rich and JSON outputs are). `--apply` ignores `--csv` (apply flow always uses the existing Rich-or-JSON output for now; this is intentional because the apply pathway emits multi-section output that doesn't flatten cleanly to CSV).

### `SUPER2` constant + fallback

```python
# cli/util.py (additions)
_GLYPH_FALLBACKS: dict[str, str] = {
    # ... 8 existing entries ...
    "\u00b2": "^2",    # SUPERSCRIPT TWO -- forecast R^2 (v1.7.33 audit, 6th strike)
}

SUPER2 = _const("\u00b2")  # "²" under TTY+UTF-8, "^2" otherwise
```

Now imported into `cli/main.py` alongside CHECK, CROSS, ARROW, LARROW, ELLIPSIS, BLOCK, WARN. The single use site at L3488:

```python

# After
console.print(f"  Rate:  {f.slope_gb_per_day:>8.3f} GB/day  (R{SUPER2}={f.fit_r_squared:.3f})")
```

## Authoritative-principle catches

### Catch #1: forecast R² (caught by automation, would have crashed in production)

The v1.7.32 lint test fired the moment U+00B2 entered `_GLYPH_FALLBACKS`. The failure message gave the exact file, line, codepoint, and offending source string. The fix was a one-line substitution.

**Without the v1.7.32 lint**: this glyph would have remained latent until someone redirected forecast output to a file on a Windows console with cp1252 default encoding — at which point Python would have raised `UnicodeEncodeError` mid-print and aborted the command. The user would have filed a bug, spent time isolating the codepoint, and the fix would have been the same one-liner — but days later, not seconds later.

### Catch #2: misplaced CSV branch (caught by diff inspection)

When applying the forecast CSV branch, the first attempt used `oldText = "        typer.echo(json.dumps(payload, indent=2))\n        return\n"` as the anchor. This text appears in **every JSON-emitting command in main.py** — the first occurrence is in `gdrive_paths_cmd` at L2600, not `forecast_cmd` at L3405. The edit went to the wrong command.

Caught when reviewing the diff: a CSV branch referencing `forecasts` (undefined in `gdrive_paths_cmd`'s scope) had been inserted into a Google Drive command's body. Reverted via a paired edit that restored the original `gdrive_paths_cmd` text and re-applied the CSV branch to forecast using a disambiguating anchor: `for f in forecasts:\n        # Color-code by status` (unique to forecast_cmd).

**Codified as lesson #53**: when editing huge files with repeated structural idioms, include 2-3 lines of surrounding context in `oldText` to disambiguate. The cost is a slightly longer anchor; the benefit is correct placement.

## Lesson #50 status update

After this ship, the three-layer defense looks like:

```
                          ┌─────────────────────────────────────────┐
                          │ NEW codepoint enters _GLYPH_FALLBACKS  │
                          └────────────────┬────────────────────────┘
                                           │
                                           ▼
                          ┌────────────────────────────────────────┐
                          │ DANGEROUS_GLYPHS set updated in        │
                          │ test_cli_util.py                       │
                          └────────────────┬───────────────────────┘
                                           │
                          ┌────────────────┴─────────────────┐
                          ▼                                  ▼
        ┌─────────────────────────────┐         ┌─────────────────────────┐
        │ NEXT PYTEST RUN              │         │ Future code that uses   │
        │ → automatic scan of cli/     │         │ the new constant via    │
        │ → fail if codepoint found    │         │ `from cli.util import`  │
        │   in a literal anywhere      │         │ → auto-fallback safe    │
        │                              │         │   on cp1252 capture     │
        │ ✅ STRIKE #6 CAUGHT THIS WAY │         │                         │
        └─────────────────────────────┘         └─────────────────────────┘
```

The pattern is now: *discover a new dangerous codepoint → add to fallback table → lint test catches all existing uses → fix them → safe forever after.*

What that pattern protects against (as of this ship):
- `█` (BLOCK), `✓` (CHECK), `✗` (CROSS), `→` (ARROW), `←` (LARROW), `…` (ELLIPSIS), `×` (TIMES), `⚠` (WARN), `²` (SUPER2) — 9 codepoints, none of which can crash cp1252 capture in `cli/*` anymore.

What's NOT protected:
- New glyphs we haven't encountered yet (e.g., `³`, `½`, `±`, emoji, mathematical operators). Future strikes from those follow the same pattern: discover → add to table → lint catches all uses → fix.

The system is self-extending. That's what "structural defense" means.

## Limitations

- **CSV cell escaping is Python `csv` module's default** (RFC 4180, QUOTE_MINIMAL). Cells with commas, quotes, or newlines get quoted; embedded quotes get doubled. Compatible with Excel / Google Sheets / pandas / duckdb / sqlite3 import. Not tab-delimited; users wanting TSV pipe through `tr ',' '\t'` or wait for a future `--csv-dialect` flag.
- **`export-clean`'s `metadata_fields_removed` is pipe-delimited inside one cell**. Loses tabular structure for downstream tools wanting per-field columns. JSON output remains the answer for that use case; CSV optimizes for readability + Excel compatibility.
- **No CSV in `audit` / `inspect` / `lineage` / `bundles list` / `sources list`**. These predate the CSV pattern. Completing the full parity is a separate ship; the analytics-shaped commands (forecast / export-clean / tier) were the higher-value targets here.
- **`gdrive paths` deliberately has no CSV**. Single-record information dumps don't benefit from CSV. JSON suffices for programmatic consumers.
- **Lint scope unchanged from v1.7.32**: still only `src/curator/cli/`. GUI / services / vendored code can contain glyphs in docstrings / Qt widget strings without false positives. If a future feature pipes GUI strings through CLI capture, lesson #50 strike #7 will happen and the lint should widen.

## Lesson #53 (new this ship)

> **When editing huge files with repeated structural idioms (>4000 lines, multiple commands with identical N-line patterns), the `oldText` parameter in `str_replace` / `edit_file` matches the FIRST occurrence in the file, not the intended one. Always include 2-3 lines of surrounding context in the anchor. The disambiguating context should be uniquely identifiable to the target site.**

Background: this ship's bug. The pattern `"        typer.echo(json.dumps(payload, indent=2))\n        return\n"` appears in every CLI command that supports `--json` (50+ commands in main.py at this point). The first occurrence is `gdrive_paths_cmd`, but I wanted `forecast_cmd`. The edit landed in the wrong place. Caught at diff-review time, before commit. The fix: use `for f in forecasts:\n        # Color-code by status` as the anchor — unique to forecast_cmd because no other command has both a `forecasts` variable AND a color-coding comment.

Future codification options:
- **Bash helper**: a `grep -n` wrapper that warns when a candidate `oldText` matches >1 line
- **Editor lint**: pre-edit step that counts occurrences before applying the patch
- **Architectural**: split `main.py` into per-command-group modules so each file is smaller and idioms don't collide

For now, the rule is: **read your diff before committing**. The bug was caught that way this ship.
