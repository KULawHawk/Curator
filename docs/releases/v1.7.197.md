## 🐛 Test pollution fix

Running the full suite to measure dialogs baseline surfaced 3 test failures in `test_gui_models_part3_coverage.py::TestMigrationProgressTableModel`:
- `test_init_with_job_id`
- `test_set_job_id`
- `test_set_job_id_to_none_clears`

All passed in isolation but failed in full-suite order. Root cause: **`test_get_job_id_exception_returns`** in `test_gui_main_window_part3_coverage.py` (v1.7.193) used `del type(...).job_id` in its `finally` block. This **deleted** the `job_id` property from `MigrationProgressTableModel` entirely. Subsequent tests in other files that referenced `.job_id` got `AttributeError` or wrong behavior.

**Fix:** capture the original property descriptor via `cls.__dict__.get("job_id")` before replacing, restore via `cls.job_id = original` in finally. Verified: all 120 tests pass in both isolation and full-suite order.

**Test-design heuristic captured inline:** never `del` a class attribute in a test cleanup; always restore the original via `__dict__.get` before replacement. The `del` is invisible in single-file runs but leaves the class in a permanently-broken state during suite-level runs.

Not adding as a new numbered lesson — generic Python testing practice, not Curator-specific doctrine.
