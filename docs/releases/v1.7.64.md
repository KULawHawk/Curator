# v1.7.64 — Deterministic bundle membership order: ORDER BY added_at, rowid

**Date:** 2026-05-12
**Type:** Bug fix in production code (SQL ordering)
**Scope:** `src/curator/storage/repositories/bundle_repo.py` (1-line SQL change)

## TL;DR

v1.7.63 went 7/9 green; Windows Python 3.11 + 3.12 still failed on a single test where 2 bundle UUIDs came back in swapped order. **Root cause: SQLite's `CURRENT_TIMESTAMP` has second resolution. Memberships inserted in the same call get identical `added_at` values, and `ORDER BY added_at` alone is non-deterministic.** Fix: add `rowid` as secondary sort key.

## Diagnosis

```
FAILED tests/gui/test_gui_bundle_editor.py::TestBundleEditorDialog::test_edit_mode_pre_populates
    AssertionError: assert [A] == [B]  (UUID order swapped)
```

The fixture: `create_manual(member_ids=[files[0].curator_id, files[1].curator_id])` inserts both rows back-to-back. Both get the same `added_at = '2026-05-12 17:50:53'`. `ORDER BY added_at` returns them in implementation-defined order, which varies by:
- SQLite version (Win 3.11 = SQLite 3.43, Win 3.13 = SQLite 3.47, etc.)
- Page layout
- Build flags

6 of 9 CI cells happened to align with test expectations on v1.7.63. Win 3.11/3.12 didn't. **The test was always flaky** — just got lucky on most platforms.

## The fix

```python
"SELECT * FROM bundle_memberships WHERE bundle_id = ? "
"ORDER BY added_at, rowid"
```

SQLite's `rowid` is monotonic on insertion. When `added_at` ties, `rowid` falls back to user-intent order (the order rows were inserted, which matches user-provided `member_ids` list order). Idiomatic SQLite pattern for deterministic ordering.

## Catches

- **Test was always flaky.** Not a v1.7.63 regression. SQLite tie-breaking happened to align with expected order on 6 platforms but not Win 3.11/3.12.
- **Caught by the OS×Python matrix.** Single-cell setups would never have seen this. 9-cell matrix earned its keep again.
- **3 alternatives considered:**
  - Set comparison in test (papers over production bug)
  - Microsecond `added_at` via `strftime('%f')` (invasive, migration needed)
  - **`rowid` secondary sort** ← chosen (1-line, idiomatic, zero schema impact)
- **`rowid` is stable.** Every non-`WITHOUT ROWID` table has it; monotonic on insert; documented SQLite primitive.
- **No baseline needed.** Pure additive ORDER BY change. Tests that depend on order will now pass deterministically; tests that don't are unaffected.
