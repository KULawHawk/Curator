## Pattern: single-instantiation = bulk coverage

The biggest insight from Part 1: a single `CuratorMainWindow(stub_runtime)` call exercises **~hundreds of statements** because `__init__` → `_build_ui` → all 9 `_build_*_tab` methods runs as a chain. The test design exploits this — most coverage comes from instantiation; focused unit tests fill the gaps for static helpers and computed methods.

## `make_runtime_stub` factory

The full stub surface (config + 9 repos + db, all returning empty/zero defaults) is captured in a `make_runtime_stub(*, time_range, config_overrides)` factory in this Part 1 file. Parts 2-4 will import and reuse it rather than redefining 60 mocked methods (Lesson #84).

## Real-class trick for property-exception tests

To test the defensive `except Exception:` in `_update_settings_header` where `config.source_path` might raise, **`MagicMock` is insufficient** — its `__getattr__` returns child mocks regardless of any `type(...).source_path = property(...)` override. Pattern: replace `runtime.config` with a real `_RaisingConfig` class that has an actual `@property` raising `AttributeError`.

**This is a refinement on Lesson #95** (the pydantic `__dict__` bypass for validate_assignment models). Captured inline in this release notes rather than as a new numbered lesson because it's the same family of "force a property to raise" trick, just for the non-pydantic case.

**Reusable when:** any test of a defensive `except` branch around `obj.attr` access where `obj` is normally a stubbed service. The pattern works for any attribute, not just `source_path`.
