# v1.7.39 — GUI parity + streaming audit-export + detached-pytest workflow

**Date:** 2026-05-11
**Type:** Combined feature ship (GUI parity + architectural streaming + workflow fix)
**Scope:** `src/curator/gui/dialogs.py` (+60), `src/curator/cli/main.py` (+22), `src/curator/storage/repositories/audit_repo.py` (+78), 2 new test files (+600), 1 new script (+97)

## TL;DR

Three pieces in one ship:

1. **GUI parity** for two prior CLI features — `share_visibility` dropdown in `SourceAddDialog` and a "Keep metadata" checkbox in `TierDialog` close the v1.7.29 and v1.7.35 GUI gaps.
2. **Streaming audit-export** — new `AuditRepository.iter_query()` generator lets `curator audit-export` stream multi-hundred-thousand-row exports with bounded memory; closes the v1.7.31 limitation 1.
3. **Workflow fix** — new `scripts/run_pytest_detached.ps1` solves an MCP-wedge problem that had been forcing Claude Desktop restarts across multiple autonomous-mode sessions. Long pytest baselines now run via detached-spawn + sentinel-poll, keeping MCP tools responsive throughout.

## Why all three landed together

The original ship plan was just (1) GUI parity. While trying to verify it, autonomous mode hit the MCP wedge problem (3) for the third session in a row — pytest baselines (>3 min) consistently wedge `Windows-MCP:PowerShell` such that every subsequent tool call hangs at the ~4 min ceiling, requiring Claude Desktop restart. The fix had to land in this ship to enable shipping anything else in autonomous mode.

(2) was already partially in flight from a parallel session — `audit_repo.py` had `iter_query()` written and 16 tests passing. Combining all three into one tagged release was cleaner than separating an already-working unit.

## Part 2 — Streaming audit-export

### Before / after

```python
# Before v1.7.39:
entries = self.audit_repo.query(since=..., limit=1000000)   # materializes 1M rows -> ~150 MB
for e in entries:                                            # then iterates
    write(format(e))

# After v1.7.39:
entries = self.audit_repo.iter_query(since=..., limit=1000000)  # generator, yields lazily
for e in entries:                                                # bounded memory
    write(format(e))
    rows_exported += 1                                           # count as we go
```

### Architecture

```
   AuditRepository.query()  ──┐
                              ├──>  _build_query_sql_and_params()  ──>  (sql, params)
   AuditRepository.iter_query() ──┘                                          │
                                                                             v
                                                                       cursor.execute()
                                                                             │
                              ┌──────────────────────────────────────────────┘
                              v
   query():       return [row_to_entry(r) for r in cursor.fetchall()]    # materialized
   iter_query():  for r in cursor: yield row_to_entry(r)                  # streamed
```

The shared `_build_query_sql_and_params()` helper means any new filter parameter (e.g., a future `--severity` flag) added once is automatically available to both `query()` and `iter_query()`. Regression-tested by `test_build_query_sql_and_params_returns_tuple`.

### `audit_export_cmd` change

Two-line switch + a manual counter:

```python
# v1.7.39:
entries = rt.audit_repo.iter_query(since=since_dt, until=before_dt, ...)
rows_exported = 0   # generators don't support len()
...
for e in entries:
    write_row(e)
    rows_exported += 1   # incremented in every format branch
```

The audit-log entry written by `audit_export_cmd` for the export itself uses `rows_exported` rather than `len(entries)`.

## Part 3 — Detached-pytest workflow

### The problem

Across three consecutive autonomous-mode sessions, the same failure recurred:

1. Session runs `pytest tests/` (~3 min, 1500 tests) through `Windows-MCP:PowerShell`.
2. MCP's 4-min internal ceiling fires; the call times out from MCP's side.
3. The pytest python.exe is still running, still holds the PowerShell stdout/stderr pipe.
4. Every subsequent MCP tool call hangs at 4 min — MCP's pipe state is wedged.
5. Only fix: Claude Desktop restart.

Three restarts in three sessions made this the highest-priority workflow problem to solve.

### The fix

`scripts/run_pytest_detached.ps1` — a small launcher that:

1. Writes a worker `.ps1` to disk next to the requested log path.
2. Spawns the worker via `Start-Process -WindowStyle Hidden -PassThru` (no `-Wait`, no `-RedirectStandardOutput`).
3. The worker handles its OWN I/O internally via `*>` shell redirection.
4. **Critical:** no file handles cross the caller → worker boundary, so MCP's stdio pipe stays free.
5. Caller returns in <2s. Worker runs to completion in the background.
6. Worker writes `EXIT:<code>` to `<log>.done` when finished.

### Verified

End-to-end test of the detacher:

```
19:34:26  Spawn call:    returned in 1s (PID 86456)
19:34:32  Poll 1 (6s):   log shows "[ 14%]", MCP responsive
19:35:13  Poll 2 (47s):  log shows "[ 18%]", MCP responsive
19:36:17  Poll 3 (111s): log shows "[ 33%]", MCP responsive
19:37:22  Poll 4 (176s): sentinel "EXIT:0", log says "1514 passed in 169.94s"
```

**170s pytest run, zero MCP wedges.** Pattern proved.

### Usage

```powershell
# Launch:
cd C:\path\to\Curator
.\scripts\run_pytest_detached.ps1 `
    -LogPath C:\tmp\pytest.log `
    -PytestArgs @('tests/', '-q', '--tb=line')

# Poll (each call <1s):
while (-not (Test-Path "C:\tmp\pytest.log.done")) {
    Start-Sleep 15
    Get-Content C:\tmp\pytest.log -Tail 1   # progress
}
Get-Content C:\tmp\pytest.log.done   # "EXIT:0"
```

## Authoritative-principle catches

### Catch #1 — Dialog dropdowns default alphabetically, not semantically

First two `SourceAddDialog` write-path tests failed with mysterious `source is None` results. Root cause: `for stype in sorted(self._registered_types.keys())` means `gdrive` (alphabetically first) is the dropdown default. The local plugin requires only `roots`; gdrive requires `credentials_path` + `client_secrets_path`. Tests not setting either left `_on_ok_clicked` to fail validation silently and return without inserting.

Fix: `_select_local_type(dlg)` helper that explicitly picks `local` and fills its required `roots` `QPlainTextEdit`.

### Catch #2 — Modal `QMessageBox.exec()` blocks unit tests

`_action_bulk_migrate` calls `result_msg.exec()` at the end to show the post-migration summary. In headless tests this hangs indefinitely. The patch must target the instance method on the class, NOT the constructor or `Icon` attribute:

```python
patch.object(QMessageBox, "exec", return_value=QMessageBox.StandardButton.Ok)
```

### Catch #3 — MCP wedges on >3-min stdio holds

Not a code bug, but a workflow bug discovered in the wild. Three sessions in a row had to be restarted because pytest baselines wedged MCP. Codified as `scripts/run_pytest_detached.ps1` + lesson #59.

## Lessons captured

- **#57**: Dialog dropdowns default alphabetically, not semantically. Tests needing a specific selection MUST set it explicitly.
- **#58**: Modal `QMessageBox.exec()` must be patched in unit tests — patch the instance method on the class, not the constructor.
- **#59**: MCP PowerShell tool wedges on >3-min stdio holds. Workflow rule: long-duration commands use `scripts/run_pytest_detached.ps1`, never foreground MCP.

## Limitations

- **GUI source-edit dialog still missing.** v1.7.39 adds `share_visibility` to `SourceAddDialog` (creation only). Changing an EXISTING source's visibility still requires `curator sources config <id> --share-visibility X`. A right-click "Source Properties" action is queued.
- **`iter_query` count metadata loss.** Streaming can't pre-count rows. Users who want "how many would I export?" before running need a separate `query()` call. Acceptable tradeoff for the memory win.
- **Detacher is Windows + PowerShell only.** A POSIX equivalent using `nohup` + `&` is straightforward but unwritten. Curator is Windows-first.
- **No tooltip snapshot test.** The share_visibility/Keep-metadata tooltips contain operational guidance; if they drift, users see inconsistent help. Future hardening could snapshot the tooltip strings.
