## Lessons captured

**Lesson #87 — When stub patterns mature, new modules hit 100% in one pass.**

The Phase Gamma arc started rough:
- `tier.py` (v1.7.81 → v1.7.83): two ships + apex-accuracy correction
- `lineage.py` (v1.7.82–83): source refactoring + dead-code removal
- `safety.py` (v1.7.84): extensive monkeypatch infrastructure + pragma decision
- `scan.py` (v1.7.86): 8 stubs + 3 passes

By `bundle.py` (v1.7.87): **one pass, 27/27 tests passing on first run, 0.68s, no failures, no iteration, no source changes.**

The difference is **pattern maturity**. The reusable templates by this point:
- **`StubRepository`**: named methods matching service usage, in-memory dicts for state, capture lists for assertions
- **`StubPluginManager`**: `StubHooks` dataclass holding `StubHookCaller` instances; hook implementations injected via helper methods
- **`make_service()`**: kwargs-default-to-fresh-stub pattern for assembling a service with selective overrides
- **`make_file_entity()`**: factory function adapted across modules

`StubBundleRepository` was composed in minutes from the standard template. `curator_propose_bundle` + `curator_source_stat` slotted in identically to the enumerate/register hooks from scan tests.

**The doctrine consequence: each apex-accuracy ship makes the next one cheaper.** This is the bull case for not cutting the 100% corner. The alternative to enforcing 100% isn't "slightly less coverage" — it's missing the compounding investment in test infrastructure that pays off in later ships.

**Corollary for arc planning:** when starting a multi-module accuracy arc, expect the first 2-3 ships to feel disproportionately expensive. That's the design investment for the rest of the arc. By the 4th-6th ship, momentum compounds and ships become routine. Don't bail on the standard during the expensive early phase.
