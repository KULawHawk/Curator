## Headline

The `lineage_fuzzy_dup` plugin has detected pairwise NEAR_DUPLICATE edges since v0.14. What's been missing: a way to see whole *families* of versions, not just pairs. v1.7.1 closes that loop.

New `LineageService.find_version_stacks()` walks NEAR_DUPLICATE + VERSION_OF edges as connected components via union-find. New `VersionStackDialog` (Tools menu) renders each stack as a collapsible group box, sorted by stack size desc with newest file first within each stack.

Captures the "Draft_1 / Draft_Final / Draft_FINAL_v2" pattern central to Jake's RCS workflow.

---

## What's new

### `LineageService.find_version_stacks()`

```python
def find_version_stacks(
    self,
    *,
    min_confidence: float = 0.7,
    kinds: list[LineageKind] | None = None,
) -> list[list[FileEntity]]:
```

Union-find with path compression. Default `kinds=[NEAR_DUPLICATE, VERSION_OF]`. Returns list of stacks sorted by size desc; each stack sorted by `mtime` desc. Filters out deleted files; drops stacks of <2 live files.

Performance: O(E + V·α(V)) where E is edge count and V is touched-file count.

### `VersionStackDialog`

Read-only viewer accessible from **Tools → Version stacks (fuzzy)...**

- Filter row: min-confidence spinner (default 0.70 matches plugin threshold) + 2 kind checkboxes (NEAR_DUPLICATE / VERSION_OF) + Refresh button
- Scrollable container of per-stack QGroupBoxes
- Each stack rendered as 4-column table: Path / Size / Modified / Type
- Status banner shows stack count, total file count, active filter summary

---

## What v1.7.1 does NOT do

- **No Apply action.** Strictly visibility. Apply semantics (keep newest / mark canonical / bundle) need atrium-reversibility v0.1 first.
- **No live edge generation.** Reads existing `lineage_edges`. With 0 edges (current canonical state), shows "No stacks found" with a hint to run a scan with the fuzzy-dup plugin enabled.
- **No cross-source stacks.** Stacks span sources in principle but the plugin's similarity index is per-source today.

## Authoritative-principle catch rate

Caught 5 API/field-name assumptions during the build:
1. `rt.db._conn` → `rt.db.execute()` exposed directly
2. `rt.db.conn` → callable, not attribute
3. `lineage_edges.kind` → `edge_kind`
4. `lineage_edges.similarity` → `confidence`
5. `FileEntity.size_bytes` / `.name` → actual `size` / no `name` field (use `source_path`)

All probed before code shipped; zero crashes on first run.
