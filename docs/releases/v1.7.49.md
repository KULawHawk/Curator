# v1.7.49 — source_type repository-level immutability (closes E3)

**Date:** 2026-05-11
**Type:** Defense-in-depth / v1.7.40 followup
**Scope:** `src/curator/storage/repositories/source_repo.py` (+50), `tests/unit/test_source_repo_immutability.py` (+200, 11 tests)

## TL;DR

`SourceRepository.update()` now raises `ValueError` if a caller tries to change a source's `source_type`. Closes the v1.7.40 limitation where GUI immutability was enforced via a disabled combobox, but the underlying repository would happily accept type changes (which silently invalidate `config_json` against a different plugin's schema). Defense-in-depth: now ANY path through `update()` — GUI, CLI, direct repository call, plugin — is protected.

## The behavior change

| Scenario | v1.7.48 | v1.7.49 |
|---|---|---|
| `update()` with same `source_type` | succeeds | succeeds (back-compat) |
| `update()` with different `source_type` | **silently invalidates config schema** | **raises ValueError** |
| `update()` on non-existent `source_id` | silent no-op | silent no-op (preserved) |
| `update()` changing other fields | succeeds | succeeds |

## The error message

```
ValueError: source_type is immutable: cannot change 'local' -> 'gdrive'
for source 'my_archive'. The existing config_json was validated against
the 'local' plugin's schema and would not be valid for 'gdrive'. Delete
and re-insert if you need to change the plugin type.
```

Includes: both source_types, the source_id, why it's blocked, and the migration path.

## Authoritative-principle catches

- **Check fires BEFORE the SQL write.** Failed updates leave the existing row completely untouched (verified by `test_failed_update_does_not_write`).
- **Non-existent source_id preserves silent no-op.** Temptation was to also raise on "not found" but that's a separate behavior change with broader blast radius. Disciplined scope: enforce only what E3 asked for.
- **Error message is the API.** Includes both source_types, source_id, the reason ("immutable"), AND the migration path ("Delete and re-insert"). Future debuggers don't have to grep the codebase to figure out the workaround.

## No new lesson codified

Application of pre-existing lessons: defense-in-depth (#29), guards before mutations (#34), clear error messages (#41).

## Limitations

- **TOCTOU window** between `get()` and `UPDATE`. Acceptable for low-frequency single-writer source-config; harden with `BEGIN IMMEDIATE` if multi-process writers emerge.
- **`upsert()` is unaffected.** That's the INSERT OR REPLACE path; deliberately replaces the entire row.
- **`set_enabled()` bypasses the check.** Only updates `enabled` flag, doesn't take source_type, safe by construction.
- **No equivalent CLI surface today.** If one is added, the repository guard catches attempts.
