# v1.7.44 — Relative test import (second CI-caught bug)

**Date:** 2026-05-11
**Type:** Test import fix (1-line change)
**Scope:** `tests/unit/test_migration_autostrip.py` (+7 comment / -1 absolute import / +1 relative import)

## TL;DR

CI run #2 on v1.7.43 surfaced a second latent local-vs-CI mismatch: `tests/unit/test_migration_autostrip.py` did `from tests.unit.test_migration import ...` (absolute). Worked locally; failed on CI with `ModuleNotFoundError: No module named 'tests'`. Fixed with a 1-line switch to relative import.

## The catch

```
ImportError while importing test module 'D:\a\Curator\Curator\tests\unit\test_migration_autostrip.py'.
tests\unit\test_migration_autostrip.py:42: in <module>
    from tests.unit.test_migration import (
E   ModuleNotFoundError: No module named 'tests'
```

CI run: https://github.com/KULawHawk/Curator/actions/runs/25708724527

## Root cause

`tests/` has a **mixed `__init__.py` state**:

| Path | `__init__.py`? |
|---|---|
| `tests/` | **MISSING** |
| `tests/unit/` | exists |
| `tests/gui/` | MISSING |
| `tests/integration/` | exists |

With this mix, `tests` is a namespace package (Python 3.3+). Locally, pytest's rootdir handling + sys.path tricks made `from tests.unit.X` resolvable. CI's clean install path didn't have the same accidental discoverability.

## Fix

`tests/unit/test_migration_autostrip.py`:

```python
# Before (v1.7.43 and earlier):
from tests.unit.test_migration import (
    migration_runtime,
    _seed_real_file,
)

# After (v1.7.44):
from .test_migration import (
    migration_runtime,
    _seed_real_file,
)
```

Relative import works because `tests/unit/__init__.py` exists — this file lives in a real package and `.test_migration` is a valid sibling-module reference.

Audited all test files via grep; only ONE file had this pattern. Single-line fix is complete.

## Authoritative-principle catches

### Catch — namespace-package handling differs between local and CI

When pytest collects from a directory mixing `__init__.py`-having and missing subdirs, rootdir + sys.path resolution depends on:
- How the project was installed (`pip install -e .` version, setuptools version, .pth files)
- Where pytest's `rootdir` was inferred from
- Whether the parent of `tests/` is implicitly on sys.path locally

These are structurally invisible without clean-environment testing. CI is the only systematic catch.

No new lesson codified — this is lesson #61 in concrete form, second occurrence in 30 minutes.
