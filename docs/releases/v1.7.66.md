# v1.7.66 — Defensive ORDER BY hardening: secondary rowid on 13 timestamp queries

**Date:** 2026-05-12
**Type:** Defensive infrastructure hardening (SQL ordering)
**Scope:** 7 repository modules, 13 SQL statements

## TL;DR

v1.7.64 fixed ONE non-deterministic `ORDER BY <timestamp>` in `bundle_repo`. An audit of all 7 repository modules surfaced **12 more queries** with the same bug class. **This ship adds `rowid` as secondary sort across all 13 sites** — defensive hardening with zero runtime cost.

## Why this matters

SQLite's `CURRENT_TIMESTAMP` has second-level resolution. Two rows inserted in the same call get identical timestamps, making `ORDER BY <timestamp>` alone non-deterministic. Different SQLite versions (which the OS×Python matrix exposes) tie-break differently. v1.7.64 caught one instance via the Windows 3.11/3.12 cells; the others were latent flakiness waiting to be triggered.

This ship closes the bug class.

## Pattern

```sql
-- Before
ORDER BY occurred_at DESC

-- After
ORDER BY occurred_at DESC, rowid DESC
```

- Direction matches primary key
- `rowid` is SQLite's automatic monotonic row-numbering (any non-`WITHOUT ROWID` table)
- Insertion order ⇒ user-intent order when timestamps tie
- Zero performance impact
- Zero schema change

## Catches

- **v1.7.64 fixed one site; v1.7.66 closes the bug class.** Single-site fixes should consider the broader class.
- **OS×Python matrix earned its keep again.** Surfaces SQLite version divergence in tie-breaking that single-cell setups would never see.
- **`rowid` is canonical.** Universal, monotonic, no schema change, no perf hit.
- **`lineage_repo` already had 2 keys but still tied.** Float `confidence` + second-precision `detected_at` can both tie — added rowid as tertiary. "Any chain of ORDER BY keys without a guaranteed-unique column is non-deterministic."
