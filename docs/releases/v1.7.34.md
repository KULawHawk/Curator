# v1.7.34 — Pre-commit hook for lesson #50 lint (layer 4)

**Date:** 2026-05-11
**Type:** Opt-in defensive infrastructure
**Scope:** New `.githooks/` directory + `.gitattributes` (3 new files, 121 lines)

## TL;DR

A new optional git pre-commit hook (`.githooks/pre-commit`) runs the v1.7.32 lesson #50 lint test before every commit. Activated per-clone with one command: `git config core.hooksPath .githooks`. **Lesson #50 now defended at four independent layers** (code helper, pytest lint, docs, git hook). Verified live in both directions — passing commit allowed, failing commit blocked with a clear message.

## Why this matters

The v1.7.32 release notes named this exact limitation:

> **No pre-commit integration.** Developers who skip pytest before commit can still introduce a glyph. A future `.git/hooks/pre-commit` or `pre-commit.com` config would close that gap. Not urgent because most developers run pytest as part of normal development, and the test runs in ~600ms.

v1.7.34 is that "future ship." It costs nothing to add and closes the only remaining bypass route for the lesson #50 defense.

## What's new

### `.githooks/pre-commit` (56 lines)

POSIX shell script that:

1. **Locates Python**: tries `.venv/Scripts/python.exe` (Windows venv), then `.venv/bin/python` (Unix venv), then `python` on PATH, then `python3`. Exits with a helpful error if none found.
2. **Runs the lint**: invokes `pytest tests/unit/test_cli_util.py::test_no_literal_glyphs_in_cli_outside_util -q --no-header` with `QT_QPA_PLATFORM=offscreen` (so the test doesn't try to open a display when something in pytest's plugin chain probes Qt).
3. **Blocks bad commits**: if pytest exits non-zero, prints a clear failure message and exits with the same code, refusing the commit.
4. **Allows good commits**: silent on success except for pytest's last 3 lines (which include the timing — comforting feedback that the hook actually ran).

Failure output:

```
FAILED tests/unit/test_cli_util.py::test_no_literal_glyphs_in_cli_outside_util
1 failed in 0.73s

pre-commit: lesson #50 lint failed. Commit refused.
pre-commit: fix the glyph(s) flagged above, or run with --no-verify to bypass.
```

The pytest failure message above the hook's message includes file/line/codepoint/fix-suggestion via the v1.7.32 lint's structured `pytest.fail()` call. The developer sees one stack-free error block with everything needed to fix.

### `.githooks/README.md` (61 lines)

Activation + design docs:

- **One-command activation**: `git config core.hooksPath .githooks`
- **Why `.githooks/` not `.git/hooks/`**: tracked in repo (propagates to clones via `git pull`), unlike the untracked `.git/hooks/` directory
- **Bypass when needed**: `git commit --no-verify` for emergency hotfixes / broken local environments
- **Opt-in by design**: new clones don't auto-activate the hook (intentional — matches the broader "layers are independent" philosophy)
- **Adding new hooks**: docs explain how to drop a script into `.githooks/`, set the executable bit, and document it

### `.gitattributes` (4 lines)

```
.githooks/* text eol=lf
```

Without this, Git for Windows would normalize the hook's LF line endings to CRLF on checkout, breaking the shebang line. With it, the hook stays POSIX-compatible regardless of platform.

# Create a file with the ACTUAL codepoint U+2713 (not the escape sequence)
content = '"""..."""\nBAD = "' + chr(0x2713) + '"\nprint(BAD)\n'
with open(r"src\curator\cli\_hook_neg.py", "w", encoding="utf-8") as f:

## Authoritative-principle catches

### Catch: the first negative test was a false negative

The first attempt at the negative test used:

```powershell
$badContent = @'
BAD_GLYPH = "\u2713"
'@
$badContent | Out-File -Encoding utf8 "src\curator\cli\_hook_neg.py"
```

The hook DIDN'T block the commit. The lint test passed. Why?

PowerShell's `@'...'@` is a **verbatim here-string** — backslash escape sequences aren't interpreted. The string `\u2713` went to disk as 6 ASCII characters: `\`, `u`, `2`, `7`, `1`, `3`. The lint scans for the **codepoint** U+2713 (3 bytes in UTF-8: `\xe2\x9c\x93`), which wasn't actually in the file. So the lint correctly didn't fire.

The fix: use `chr(0x2713)` in Python to inject the actual codepoint, then `assert chr(0x2713) in open(path, encoding='utf-8').read()` to verify it landed.

Generalizes: any test verifying handling of a special character must verify the character is actually IN the test fixture, not just in the source code that generated the fixture. PowerShell verbatim strings, Python raw strings (`r"\u2713"`), and single-quoted shell strings all silently preserve the escape sequence rather than expanding it.

### Codified as lesson #54

> **When testing that a system handles a specific Unicode codepoint, ensure the codepoint actually hits disk in the test fixture. PowerShell verbatim strings, Python raw strings, and shell single-quoted strings all preserve the literal escape-sequence ASCII rather than expanding it. Use `chr(0x2713)` / `"\u2713"` (non-raw, non-verbatim) / `python -c "print(chr(0x2713))"` to inject codepoints reliably. Verify by reading the file back and confirming the codepoint is present: `chr(0x2713) in open(path, encoding='utf-8').read()` should be True.**

This lesson goes in the corpus alongside the others. Cost of the bug: ~3 minutes (write test → see it pass when it should fail → realize what's wrong → rewrite test correctly → see it fail correctly). Cheap because diff inspection caught it immediately.

## Limitations

- **Opt-in, not auto-applied.** New clones don't activate the hook automatically. Must run `git config core.hooksPath .githooks` once per clone. A future `scripts/setup_dev_env.py` could automate dev-environment setup (activate hook + install pre-commit-cli + check Python version + ...).
- **POSIX shell only.** Works under Git for Windows (bundled `sh.exe`), macOS, Linux. Doesn't work in environments lacking POSIX shell support. A PowerShell-native variant could be added if needed.
- **Single lint test.** The hook runs only `test_no_literal_glyphs_in_cli_outside_util`. If future code-quality lints are added (missing docstrings, import order, type stubs), the hook needs to expand. The ~600ms budget supports 5-6 small fast tests before commit lag becomes noticeable.
- **Bypassable with `--no-verify`.** Intentional escape hatch for emergencies. Means the hook is a *strong nudge*, not an *absolute guarantee*. The pytest layer remains the hard line — even a `--no-verify` commit will fail when someone runs `pytest tests/` on the resulting state.

## Future enhancements

If lesson #50 reasserts itself or the lint expands:

1. **Pre-push hook**: catches commits made elsewhere (e.g. on a CI box) that didn't run through the local pre-commit. Cheap to add: copy the existing hook to `.githooks/pre-push` and adjust the exit-code semantics.
2. **Expanded lint scope**: include `src/curator/gui/` when text from GUI ever flows to a CLI capture path (lesson #50 strike #7 scenario). The lint test's `cli_dir` variable would change.
3. **pre-commit.com framework**: structured hook config with reusable hook libraries. Heavier, but standard in the Python ecosystem. Could integrate alongside or replace this shell hook.
4. **Auto-fix mode**: if the lint detects a literal glyph, automatically suggest (or apply) the import + substitution. Cute but risky; better as a manual `curator-dev fix-glyphs` script if it ever gets built.
