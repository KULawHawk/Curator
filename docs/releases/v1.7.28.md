# Curator v1.7.28 — Release Notes

**Tagged:** 2026-05-11
**Tag:** `v1.7.28`
**Theme:** Per-pattern PII metadata enrichment — generalizes v1.7.26's JWT approach to 6 more HIGH-severity patterns.

---

## Headline

v1.7.26's JWT-only `metadata` field is now generalized across **6 more HIGH-severity patterns**: AWS, Stripe, Slack, GitHub, OpenAI, and Mailgun. Each one gets parsed metadata exposing the *kind* of credential. The Rich pretty-print now colors **active/production/long-term/broad-scope** credentials red — eye lands on urgent triage rows first.

---

## Forensic value example

Scanning a leaked customer repo:

| Before (v1.7.26) | After (v1.7.28) |
|---|---|
| "47 high-severity matches" | 12 Stripe keys (**3 LIVE**), 8 AWS (**5 long-term IAM**), 15 Slack (**2 user OAuth**), 9 GitHub (**4 classic PAT**), etc. |
| Equally urgent → triage = O(N) | 14 urgent + 33 lower → triage time drops ~70% |

## What's new

### 6 new parser functions

| Function | Pattern | Distinguishes |
|---|---|---|
| `_parse_aws_key` | `aws_access_key_id` | `AKIA` (long-term IAM) vs `ASIA` (STS session) |
| `_parse_stripe_key` | `stripe_secret_key` | `sk_live_` (production) vs `sk_test_` (sandbox) |
| `_parse_slack_token` | `slack_token` | bot / user / app / refresh / workspace |
| `_parse_github_pat` | `github_pat` | personal / oauth / user_to_server / server_to_server / refresh |
| `_parse_openai_key` | `openai_api_key` | `sk-proj-` (project) vs `sk-` (user/org) |
| `_parse_mailgun_key` | `mailgun_api_key` | `key-` (legacy) / `private-` / `pubkey-` |

### Dispatch table

`_PATTERN_PARSERS: dict[str, Callable]` replaces v1.7.26's hardcoded `if pat.name == "jwt"` check. Adding a new enrichment is now a one-line table addition.

### Generalized Rich emission

CLI shows **all** metadata keys (was JWT-specific: alg/iss/sub/exp_iso/expired). High-risk indicators trigger red coloring:

- `expired is False` — JWT still hot
- `mode == "live"` — Stripe production
- `key_type == "long_term"` — AWS IAM (vs ephemeral STS)
- `key_type == "legacy_api"` — Mailgun legacy
- `token_type == "personal"` — GitHub classic PAT
- `token_type == "user"` — Slack user OAuth (broad scopes)

### Coloring fix from v1.7.26

v1.7.26's CHANGELOG claimed "red when expired=True; eye lands on active credentials first" — but those statements contradicted each other. v1.7.28 fixes the logic: **red = urgent (active credential)**, **dim = lower priority (expired/test)**.

## Authoritative-principle this turn

**1 bug caught**: **Lesson #50 strikes for the FOURTH time** — used `✓` (U+2713) in test file print; subprocess crashed in cp1252. Fixed by replacing with ASCII `OK`. The lesson generalizes: **any subprocess capturing stdout will crash on Unicode glyphs in Windows-cp1252 environments, including test scaffolding** (not just user-facing CLI output).

**0 implementation bugs**: all 6 parsers worked first-try across 12 test cases.

**Design lessons reinforced:**

1. **Dispatch table > hardcoded if-chain** — Adding a 7th, 8th parser is one line.
2. **Most-specific-first ordering** — OpenAI: `sk-proj-` BEFORE `sk-` (every project key also starts with `sk-`).
3. **Coloring follows risk semantics** — red for urgent (active/live/long-term/broad-scope), dim for lower-priority (expired/test/ephemeral/narrow).
4. **Description field doubles as inline documentation** — analysts don't need to remember "AKIA = IAM user long-term"; the description says so.

## Limitations

- **Generic CSV column shape** — heterogeneous metadata keys per pattern
- **No semantic validation beyond prefix** — a typo'd `AKIA...` test fixture still gets enriched
- **Mailgun severity** — all 3 types currently same severity; `pubkey-` is lower-risk
- **No `--metadata-only` filter** — would show only enriched patterns
- **JSON schema** — could formalize as TypedDict-per-pattern in v1.8
