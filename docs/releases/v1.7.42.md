## TL;DR

CI exists now. Every push to `main` and every pull request runs the full pytest baseline on a Windows runner with Python 3.13. The README shows a live status badge.

## Why this matters

For 41 consecutive ships, "main is green" depended entirely on me running pytest locally before pushing. No automatic protection. If I skipped that step (or if a flaky test passed locally but would fail in a clean environment), main could be silently broken.

In the bulletproof-live analysis at v1.7.41, "no CI" was the single biggest invisible risk. The user called it out directly. v1.7.42 closes the gap.

## What's new

### `.github/workflows/test.yml`

Single-job workflow. On every `push: main` and `pull_request: main`:

1. Checkout
2. Setup Python 3.13 with pip cache (keyed on `pyproject.toml`)
3. `pip install -e ".[all]"` (full extras profile)
4. `pytest tests/ -q --tb=line --timeout=120` with `QT_QPA_PLATFORM=offscreen`

### Design choices (each documented inline)

| Choice | Reason |
|---|---|
| `windows-latest` only | Curator is Windows-first; matches dev exactly. Cross-platform matrix is a future ship. |
| Python 3.13 only | Matches local dev. `[3.11, 3.12, 3.13]` matrix is a future ship. |
| `[all]` extras | Matches `setup_dev_env.py --profile full`; same install path real contributors use. |
| `--timeout=120` per-test | Catches hangs without killing the whole run. |
| 15-min job timeout | 5x headroom over typical 3-min local baseline. |
| `concurrency.cancel-in-progress: true` | Saves CI minutes on rapid fix-up pushes. |
| pip cache on `pyproject.toml` | Speeds up repeat runs. |

## Authoritative-principle catches

### Catch — I was prioritizing feature ships over infrastructure for 41 ships

The user's pointed question ("why are we not doing this?") surfaced that I was picking shipable feature work because features have clear end states ("did the dialog work?") while CI requires a one-time foundational decision I never made. Bad prioritization. Once stated plainly, the pivot was immediate.

### Design discipline — single OS, single Python for v1

Tempting to build a 9-cell matrix on day one (`[ubuntu, windows, macos] × [3.11, 3.12, 3.13]`). Each cell adds runner-minutes + complexity. 80/20 rule: cover dev environment first, expand when proven needed. Documented in YAML so future contributors know it's intentional.

## Lessons captured

- **#61: Infrastructure ships beat feature ships when missing.** 30 minutes on YAML that protects every subsequent ship is dramatically higher-leverage than 30 minutes on another feature. Signal: "green baseline" is a claim that depends on a single human running a single command. When you notice that, drop the feature and ship the infrastructure. The user surfacing this was the prompt; the lesson is to look for it pre-prompt next time.

## Limitations

- **No Linux/macOS coverage.** Future ship if Curator goes cross-platform.
- **No Python 3.11 / 3.12 matrix.** pyproject claims `>=3.11`; only 3.13 exercised.
- **No coverage reporting.** `pytest-cov` + `codecov` is a future ship.
- **Flaky migration test occasionally fails CI** at the same rate it fails locally (~5-10%). Manual CI re-run is the workaround; real fix is A1 from the bulletproof-live backlog.
- **No CI-side lint step.** lesson #50 lint runs as part of pytest (defense in depth via `.githooks/pre-commit` + the pytest test). Separate explicit step would be redundant.
- **No release artifact builds.** `git tag v1.7.42` doesn't publish to PyPI. Future release engineering ship.
- **CI runner minutes billed.** Free tier is 2000 min/month; ~400 pushes/month before limit. Current pace well under that.
