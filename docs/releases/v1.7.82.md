## Lessons captured

**Lesson #68 — "Pure service" doesn't mean "pluggy-free."**
I told you lineage.py had "no pluggy entanglement" when proposing it vs bundle.py. That was wrong — lineage uses pluggy for `_run_detectors`. I skimmed for `pluggy.PluginManager` in the constructor without tracing the method bodies. **Going forward: trace method bodies before recommending a test target, not just the constructor signature.**

**Lesson #69 — Coverage estimates should be ranges, not point estimates.**
I claimed "90%+" in the tradeoff table. First pass landed at 73%; reaching 94% required a second pass with FuzzyIndex stubs. Future estimates: ranges ("70-90% depending on stub scaffolding") with an explicit "measure-then-decide-whether-to-push" step. v1.7.81's 99% on tier.py was a lucky shape, not a baseline expectation.

**Lesson #70 — Stubs test "what the service does with results," not "whether the query was constructed correctly."**
`StubFileRepository.query()` returns `_query_results` regardless of the FileQuery filters passed in. That tests the service's *use* of results, not the *construction* of the query. Integration tests catch the latter. Document this boundary in stub docstrings to prevent future contributors from over-testing in the unit layer.
