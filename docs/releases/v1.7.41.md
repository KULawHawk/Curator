## TL;DR

New `scripts/setup_dev_env.py` takes a fresh clone to a working dev install + smoke-tested baseline in one command. Three profiles (`minimal` / `standard` / `full`) cover most contributor needs; `--dry-run`, `--force`, and `--no-smoke` flags handle edge cases.

## The five steps

| Step | What | Failure exit |
|---|---|---|
| 1 | Python version check (`>= 3.11`) | 2 |
| 2 | Project root resolution | 3 |
| 3 | Venv creation | 4 |
| 4 | `pip install -e ".[<profile>]"` | 5 |
| 5 | Smoke test (collect-only) | warn-only |

## Profile mapping

| `--profile` | Extras |
|---|---|
| `minimal` | `[dev]` (~6 packages) |
| `standard` | `[dev,beta,organize]` (~20 packages) |
| `full` (default) | `[all]` (~40+ packages) |

## Lessons captured

- **#60**: Polling MCP calls themselves should be short (<60s). Even with detached worker spawns (lesson #59), individual MCP tool calls that hold open for > ~2 min can wedge the MCP server. Multiple short poll calls beat one long one.

## Limitations

- Smoke test is collect-only (catches import errors, not subtle install issues)
- `--force` doesn't `rm -rf .venv` first
- Profile `standard` doesn't include GUI deps
- Flaky migration test (`test_abort_during_run_marks_cancelled`) is pre-existing
- No `uv` integration
