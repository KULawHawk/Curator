# v1.7.38 — Clean error for invalid `--csv-dialect` (closes v1.7.37 limitation)

**Date:** 2026-05-11
**Type:** UX polish (rare-path error message)
**Scope:** `src/curator/cli/main.py` (+30 lines), `tests/integration/test_cli_csv_dialect.py` (+71 lines)

## TL;DR

New `_check_csv_dialect()` helper in `cli/main.py` catches the v1.7.37 `--csv-dialect xyz` typo case at the CLI layer and surfaces a clean typer-style error message instead of letting `build_csv_writer`'s `ValueError` propagate as a Rich traceback. Two-layer validation preserved: helper-side defends library callers; CLI-side polishes the user experience.

## Why this matters

v1.7.37's TSV dialect ship documented a limitation: `--csv-dialect xyz` produced a Python traceback rather than a clean CLI error. The bug was rare-path polish, not a correctness issue — valid inputs always worked correctly. But traceback-on-typo is a poor user experience and made the CLI feel half-finished. v1.7.38 closes that polish gap with ~30 lines of code + 3 tests.

## What's new

### `_check_csv_dialect(rt, dialect)` helper in main.py (~16 lines)

```python
def _check_csv_dialect(rt: CuratorRuntime, dialect: str) -> None:
    """Validate the --csv-dialect value at the CLI layer.
    
    v1.7.38: keeps the deeper build_csv_writer ValueError behavior
    intact (defensive coding at the helper layer stays useful for
    library callers) while ensuring CLI users see a typer-style error
    instead of a Rich traceback.
    
    The two-layer validation is intentional:
      * CLI side: clean user-facing error for typical typo case
      * Helper side: defends programmatic misuse from code
    
    Both raise; CLI raises typer.Exit(1), helper raises ValueError.
    """
    if dialect not in ("csv", "tsv"):
        raise _err_exit(
            rt,
            f"--csv-dialect must be 'csv' or 'tsv'; got {dialect!r}",
        )
```

## Authoritative-principle catches

### Design choice: layered validation

Rather than removing the helper-side `ValueError` defense, kept it AND added the CLI-side helper. The two-layer approach has clear value:

- **CLI users**: polished error path (clean message, no traceback, exit 1)
- **Library callers** (anyone using `build_csv_writer` programmatically): still get the defensive ValueError
- **Testability**: helper's behavior testable in isolation (`test_helper_validation_still_raises_valueerror`), separately from CLI surface (`test_invalid_csv_dialect_*`)

The pattern is reusable: anywhere a util helper raises a clean exception, the CLI layer can add a thin validation step that surfaces it as `typer.Exit`. We already follow this pattern in audit-export's `--format` validation, so v1.7.38 makes it consistent for `--csv-dialect` too.

### No new lesson

The pattern (layered validation: helper raises, CLI catches) is well-established CLI design. Not novel enough to codify as a lesson; the existing pattern in audit-export's `--format` validation predates this ship.

### 0 production bugs caught

Implementation was straightforward. No surprises during smoke-testing.

## Limitations

- **Helper still uses string comparison, not Enum.** `_check_csv_dialect` validates against the tuple `("csv", "tsv")` rather than using a typed Enum. Tradeoff: simpler typer Option declaration (no extra Enum import or callback), at the cost of a magic-string list maintained in two places. If a third dialect is ever added, both spots need updating. Acceptable for the current size (2 dialects); reconsider if it grows.

- **No tab-completion for the dialect value.** Typer can offer shell completion for fixed sets via Enum-backed options. v1.7.38 stayed with `str` for the simpler declaration. Users wanting tab completion can run `--help`.

- **The check fires only when `--csv` is also passed.** Without `--csv`, the `--csv-dialect` value is unused and unvalidated. This is correct (no behavior to validate) but means typos like `--csv-dialect tcv` (no `--csv`) silently pass. Acceptable since `--csv-dialect` alone has no effect anyway.
