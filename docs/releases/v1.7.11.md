# Curator v1.7.11 — Release Notes

**Tagged:** 2026-05-11
**Tag:** `v1.7.11`
**Theme:** T-B04 v3 — 3 more API key patterns (Google, Stripe, OpenAI).

---

## Headline

`curator scan-pii` gains 3 more HIGH-severity API key patterns:
- **Google API key** (Maps/Firebase/YouTube/GCP)
- **Stripe secret key** (live + test mode)
- **OpenAI API key** (legacy `sk-` + new `sk-proj-`)

All prefix-distinct so they don't collide with each other or with the v1.7.10 set. Total patterns: **11** (was 8).

---

## What's new

3 new default patterns (uses the validator hook shipped in v1.7.10):

| Name | Severity | Regex |
|---|---|---|
| `google_api_key` | HIGH | `\bAIza[0-9A-Za-z\-_]{35}\b` |
| `stripe_secret_key` | HIGH | `\bsk_(?:live\|test)_[A-Za-z0-9]{24,}\b` |
| `openai_api_key` | HIGH | `\bsk-(?:proj-)?[A-Za-z0-9_\-]{20,}\b` |

### Collision design

Stripe uses `sk_` (underscore); OpenAI uses `sk-` (dash). The patterns are mutually exclusive by their prefix character. Test 4 of the new suite specifically verifies that `sk_live_X` matches Stripe only and `sk-Y` matches OpenAI only.

### What's NOT flagged

- **Stripe publishable keys** (`pk_live_`, `pk_test_`) — intentionally public, not a secret. Test 2 verifies these are not matched.
- **Google `gsk_` or other prefixes** — only the `AIza` prefix is the standard.
- **Truncated or malformed tokens** — strict length requirements via `{35}` etc.

## Authoritative-principle catches

**2 test-data length bugs caught immediately by assert statements:**

Initial test data was hand-typed and miscounted:
- `"ABCDEFGHIJKLMNOPQRSTUVWXYZ_012345678"` looks like 35 chars but is 36 (`_012345678` is 10 chars not 9)
- Test 5's Google key construction had a typo producing 40 chars instead of 39

Both caught by `assert len(...) == 35` running BEFORE the regex test, so failures showed clearly as test-data bugs not regex bugs.

**Lesson #45 logged**: when test data must match an exact-length regex, construct it programmatically (`"a" * 35`) instead of typing the body manually. Add length assertions BEFORE the regex test runs.

**0 regex bugs caught** — the 3 new patterns worked first-try once test data was correct. The "unique prefix + character class" pattern continues to be reliable for API tokens with documented formats.

## Limitations / next steps

- **No Mailgun key** — older `key-` format being deprecated; `private-`/`pubkey-` not yet stable
- **No Discord bot token** — 3-segment dot format needs careful tuning; deferred
- **No Twilio account SID** — distinctive prefix but typically needs contextual detection
- **No GCP service account JSON** — multi-line, needs different scanner approach
- **No Conclave hookspec for custom validators** — still on v1.8 list
