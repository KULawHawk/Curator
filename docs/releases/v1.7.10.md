## What's new

### Validator hook on PIIPattern

```python
@dataclass(frozen=True)
class PIIPattern:
    name: str
    pattern: re.Pattern[str]
    severity: PIISeverity
    description: str
    validator: Callable[[str], bool] | None = None  # NEW

    def is_valid(self, value: str) -> bool:
        if self.validator is None:
            return True
        try:
            return bool(self.validator(value))
        except Exception:
            return False  # safe-by-default
```

When a validator is set, regex matches that fail the validator are silently dropped from the scan. Pluggable design — adding more validators later doesn't require core-scanner changes.

### Two validator implementations shipped

```python
_luhn_valid("4532015112830366")  # True (real Visa test card)
_luhn_valid("1234567890123456")  # False (random; checksum invalid)

_ipv4_valid("10.0.0.5")           # True
_ipv4_valid("999.888.777.666")    # False (octets > 255)
_ipv4_valid("192.168.001.001")    # False (leading zeros = typo)
```

## Authoritative-principle catches

**1 regex greedy-match bug caught immediately by Test 2:**
- Original `credit_card` regex `\b(?:\d[ -]?){13,16}\b` greedily consumed the trailing space (the inner optional `[ -]?` matched the post-card space, and `\b` happily anchored at the space→word transition). `matched_text == "4532015112830366 "` (17 chars), which broke last-4 redaction.
- **Fix**: tightened to `\b(?:\d[ -]?){12,15}\d\b` — 12-15 reps of "digit + optional separator" plus one **required** final digit. Same total digit range; forces match to end on a digit.
- **Lesson #44 logged**: when an inner regex element ends with an optional separator and the outer `\b` anchors at the end, the engine will greedily consume the separator if it leads to a valid boundary. **Always anchor on a required digit at both ends** when matching "digits with optional separators between them."

## Limitations / next steps

- **No IPv6** — regex is non-trivial (`::`, mixed notation, zone IDs); deferred.
- **No private vs public IP distinction** — could separate `10.0.0.0/8` etc. as INFO-only.
- **No AWS Secret Access Key pattern** — secret keys are 40 chars of base64 with no distinctive prefix; high FP rate.
- **No `xoxc-` Slack variant** — browser-extracted tokens have slightly different format.
- **No Google API key / Stripe / Mailgun / Discord patterns** — could batch-add in v1.7.11 if demand surfaces.
- **No Conclave hookspec for custom validators** — `validator` field is Python-callable internally but not yet pluggable from outside the module.
