## Lesson captured

**Lesson #89 — Scope plans need revision after sub-ship 1. Coverage estimates calibrate to actual delta.**

The Migration Phase Gamma scope plan (v1.7.88) estimated each sub-ship's coverage gain based on target cluster size. v1.7.89's estimate: ~72-75%. Actual: 68.18%. **The estimate was off by a factor of 3-5x.**

Why: I assumed "Plan() edges + Apply() control flow" would naturally pull in incidental adjacent lines. It didn't. Defensive branches scattered across a 3000+ line file are smaller-grain than visual line-range hints suggest. A `547-554` range looks like 8 lines but in coverage terms it's often 1 logical branch + 5 statements that become covered or not as a unit.

**The lesson isn't "don't write scope plans"** — the plan still produced a sound multi-ship structure. The lesson is: **after the first sub-ship lands, REVISE the remaining estimates based on actuals**, and bake the revision into the plan document.

Two errors avoided by surfacing the calibration:
1. **Continuing to plan against stale estimates** — if I'd proceeded to v1.7.90 expecting ~78%, I'd have set wrong success criteria and either overengineered tests to chase the number or shipped feeling like I underdelivered.
2. **Hiding the miss** — "landed at 68% when plan said 75%" sounds like a failure unless surfaced as deliberate calibration. The CHANGELOG, scope-plan tracker, and this release note now own the miss, the lesson, and the revised estimates.

**General rule for multi-ship arcs:** the first sub-ship is a calibration data point. After it lands, update the plan with actual cost-per-coverage-percent, revise downstream estimates, and document the revision. The arc target (100% here) is unchanged; only per-sub-ship deltas recalibrate.
