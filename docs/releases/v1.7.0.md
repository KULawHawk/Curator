# Curator v1.7.0 — Release Notes

**Tagged:** 2026-05-11
**Commit:** `a41840d`
**Tag:** `v1.7.0`
**Theme:** GUI parity for the v1.6 CLI surface. Tools menu has zero placeholders.

---

## Headline

The PySide6 desktop GUI now covers the full v1.6 CLI surface for **scan / cleanup / find duplicates / health check / sources management / audit log review**. All five v1.6.2 Tools-menu placeholders have graduated to real native dialogs (or a tab pivot). One new tab — **Sources** — was added.

You can now do everything you'd previously have done via the CLI (`curator scan`, `curator cleanup`, `curator group`, `curator sources ...`, `curator health-check`, `curator audit`) entirely from inside the GUI, with progress bars, threaded execution, and per-row context-menu actions.

---

## What's new

### 5 new native dialogs

| Dialog | Replaces CLI | Background thread? |
|---|---|---|
| **`ScanDialog`** | `curator scan <folder>` | ✅ `ScanWorker` QThread |
| **`GroupDialog`** | `curator group --apply` | ✅ Two-phase: `GroupFindWorker` + `GroupApplyWorker` |
| **`CleanupDialog`** | `curator cleanup junk-files / empty-dirs / broken-symlinks` | ✅ `CleanupFindWorker` + `CleanupApplyWorker` |
| **`SourceAddDialog`** | `curator sources add` | ❌ (single DB insert, microseconds) |
| **`HealthCheckDialog`** | `curator health-check` | ❌ (8 sections, 22 checks, ~4 seconds) |

### 1 new tab: **Sources** (9th tab)

Live table showing every registered source with per-row context-menu actions (Enable / Disable / Remove). Top toolbar has `+ Add source...` (opens `SourceAddDialog`) and `Refresh`. Disabled sources render gray. Remove gracefully refuses if files still reference the source (SQL `ON DELETE RESTRICT`) and suggests Disable instead.

### Audit Log filter UI

The Audit Log tab now has a 6-control filter toolbar backed by `AuditRepository.query()`'s native filter kwargs:

| Kwarg | UI widget |
|---|---|
| `since` | "N hr ago" hour-spinner (0 = no limit) |
| `actor` | Dropdown (auto-populated from DB) |
| `action` | Dropdown (auto-populated from DB) |
| `entity_type` | Dropdown (auto-populated from DB) |
| `entity_id` | Text input (exact match) |

Apply / Clear / ↻-refresh-dropdowns buttons. Status label shows row count + active filters.

## Authoritative-source-first principle

This release codified Lesson 22-24 from the v1.7-alpha sequence:

> **When the user asks for a comprehensive list, count, or inventory of anything that has an authoritative source, the FIRST tool call must produce that authoritative answer. My mental model and probes generate hypotheses; the authoritative source produces deliverables.**

Authoritative sources used throughout the alpha sequence:

| Question | Authoritative source |
|---|---|
| What files exist in the repo? | `git ls-files` (NOT directory walks) |
| What methods does this class have? | `inspect.getmembers(cls, predicate=inspect.isfunction)` |
| What signature does this method have? | `inspect.signature(method)` |
| What fields does this Pydantic model have? | `cls.model_fields` |
| How many rows in this table? | `SELECT COUNT(*) FROM table` (NOT inferring from samples) |
| Which plugins are registered? | `pm.list_name_plugin()` / hookspec results |
| Which CLI commands exist? | Typer app's `registered_commands` walk |

**Trigger:** User word "all" / "every" / "complete" / "100%" → authoritative source first, full stop.

This principle caught **7+ API assumption errors** during the v1.7-alpha dialog builds before they shipped, plus 3 broken pytest tests with hard-coded tab counts that `pytest -x` caught immediately on first run after the change.
