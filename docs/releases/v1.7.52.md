## TL;DR

New `curator mcp cleanup-orphans` CLI command finds (and optionally kills) orphaned `curator-mcp.exe` processes. Dry-run by default; opt-in to actually kill via `--kill --yes`. Supports `--json` for automation. Closes A4 from the bulletproof-live backlog.

## The annoyance this closes

When an MCP client (Claude Desktop, etc.) crashes or is force-quit, its spawned `curator-mcp.exe` subprocess often outlives it. Each orphan:
- Holds an open SQLite handle to the Curator DB (eventually exhausts handles)
- Consumes ~30-50 MB RAM
- Pollutes Task Manager
- Can race new MCP server instances for the same DB lock

Manually killing them via Task Manager is tedious. Now:

```
# List orphans (safe dry-run)
$ curator mcp cleanup-orphans

# Kill orphans with confirmation
$ curator mcp cleanup-orphans --kill

# Kill orphans for automation
$ curator mcp cleanup-orphans --kill --yes

# JSON for scripting
$ curator --json mcp cleanup-orphans
```

## Design decisions

- **psutil required** — gracefully reports install command via `[organize]` or `[all]` extras
- **Dry-run by default** — destructive operations require explicit `--kill`
- **Confirm by default** — even with `--kill`, prompts unless `--yes`
- **Cross-platform** — matches both `curator-mcp.exe` (Windows) and `curator-mcp` (POSIX)
- **Graceful kill** — terminate → 3s wait → kill if needed
- **Idempotent** — `psutil.NoSuchProcess` during kill counts as success
- **`--json --kill` requires `--yes`** — no interactive prompt available; exits 2 if not provided

## Authoritative-principle catches

- **Click removed `mix_stderr` from CliRunner.** Adapted tests to check `result.output` (combined) instead of separate streams.
- **Mocked `_enumerate_curator_mcp_processes`, not `psutil.process_iter`.** Mocking one function up the stack is simpler than faking psutil's lazy proxy objects.
- **`--json --kill` without `--yes` is intentionally an error.** Destructive operations should never silently default to yes.
- **`psutil.NoSuchProcess` during kill counts as success.** Race condition between enumeration and kill is idempotent; treating it as failure would produce spurious exit-1 on reruns.

## Limitations

- No automatic cleanup on Curator startup (could wire later)
- psutil requirement excludes `[minimal]` profile (clear install message)
- No Docker container detection
- Tests mock the process layer (real psutil only via smoke test)
