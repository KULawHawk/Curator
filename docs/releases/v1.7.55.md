## Why this matters

v1.7.51's coverage baseline showed **zero existing test files referenced `pii_scanner` by name**. The 22% was incidental coverage from other tests transitively importing the module. PII detection is forensically sensitive — false negatives = leaked sensitive data; false positives = signal drowning. You want very high test coverage on this file, not low coverage.

## Authoritative-principle catches

- **One test had a bug** — Google API key needed exactly 35 chars after `AIza`; I gave 36. Fixed in one line. Code verified test as much as test verified code.
- **Imported underscore helpers deliberately** — they're documented extension points. Locking the contract is more valuable than nominal privacy.
- **JWT test helper doesn't sign cryptographically** — `_parse_jwt` never verifies signatures; testing with real signed JWTs would add no coverage.
- **Non-UTF8 file path tested** — bad bytes around valid SSN; SSN is still found (v1.7.6 design intent verified).
- **5 unmissed statements accepted** — pure defensive code; mocking stdlib internals to reach 100% is disproportionate.
