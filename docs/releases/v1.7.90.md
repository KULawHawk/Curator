## Lesson captured

**Lesson #90 — Service-orchestration tests must trace the data flow.**

Both of my initial test attempts failed because I didn't trace the orchestration carefully:

1. **Assert-on-original-vs-copy:** `apply()` creates a *fresh copy* of each move into `report.moves` and mutates the copy. The original move stays unmutated. My first 13 assertions all referenced the original and failed with `outcome=None`. Fix: assert against `report.moves[0]`.

2. **Gate-bypass:** `apply()` runs three gates before `_execute_one_same_source` — REFUSE check, DB-guard, collision-resolve. Pre-creating dst to trigger the OSError-during-copy path made Gate 3 (collision) intercept with the default `skip` mode and return `SKIPPED_COLLISION` before `_execute_one` ran at all. Fix: don't pre-create dst; make the fake `copy2` create a partial dst mid-execution and *then* raise.

**General rule:** before writing a surgical unit test that targets a specific branch in an orchestration method, read the orchestration's control flow from entry to that branch. Note every short-circuit, every copy, every mutation. **Lesson #82 said "the question is stub infrastructure, not feasibility" — the corollary is that stub infrastructure design must match the data flow, not just the dependency graph.**

Both errors were caught quickly because the existing 178 integration tests gave reasonable coverage of the happy paths. The targeted unit tests are filling gaps in *error paths and policy variants* — exactly where the orchestration's gates and copies matter most.
