## Headline

**Files classified as `status='vital'` are now safe from accidental trashing.**

No Curator-side source code changes — all behavior comes from atrium-safety v0.4.0's new `curator_pre_trash` hookimpl using Curator's existing `curator_pre_trash` hookspec. The user-facing behavior shift is felt through Curator, so a Curator version-bump documents it; the actual code lives in the plugin (where it belongs architecturally).

This is the **first cross-repo coordinated release** in the Curator + plugins constellation.

---

## What's new (via atrium-safety v0.4.0)

### New hookimpl: `curator_pre_trash`

Returns `ConfirmationResult(allow=False, ...)` for:
- Files with `status='vital'` AND no `expires_at` set, OR
- Files with `status='vital'` AND `expires_at` in the future (retention horizon still active)

ALLOWS (with audit) when:
- Files with `status='vital'` AND `expires_at` in the past (retention period elapsed)

Files in any other status bucket (`active` / `provisional` / `junk`) behave exactly as before — no veto.

### Audit events

| Action | Fires when |
|---|---|
| `compliance.retention_veto` | Trash blocked by retention enforcement |
| `compliance.retention_allow` | Retention horizon allows previously-vital file to be trashed |

Both use `actor='curatorplug.atrium_safety'`, `entity_type='file'`, `entity_id=<curator_id>`. Best-effort emission — silently no-ops if plugin doesn't have a `pm` reference yet.

## Override paths (for users)

If you legitimately need to trash a vital file:

```powershell
# Option 1: Reclassify (most common)
curator status set /path/to/file.txt active

# Option 2: Force-expire retention
curator status set /path/to/file.txt vital --expires-in-days -1
```

Each override is auto-audit-logged via `cli.status` (action: `file.status_change`). The audit trail captures both the override and the subsequent trash operation.

---

## Graceful degradation

The hookimpl handles three degradation cases cleanly:

1. **Pre-v1.7.3 Curator (no migration 003 applied)** — `FileEntity` lacks `.status` attribute. Treated as `active`. No veto fires.
2. **Plugin not yet `curator_plugin_init`'d** — vetoes still fire (no `pm` access needed for the check itself). Audit emission silently skipped.
3. **Pre-v1.1.3 Curator (no `curator_audit_event` hookspec)** — vetoes still fire. Audit emission silently skipped.

In every case, the plugin fails open or fails closed in the safest direction.
