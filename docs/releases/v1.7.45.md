## Class A root cause

CI's pytest output for one of the 3 tests:
```
assert '--keep-under' in '\x1b[1m \x1b[0m\x1b[1;36m--keep-under\x1b[0m...'
```

Rich is wrapping option names in ANSI bold/cyan, BUT it also inserts a leading `\x1b[1m \x1b[0m` (bold-space-reset) between leading whitespace and the option text. That breaks the literal substring match.

Locally, Rich auto-detects pytest's captured stdout as non-TTY and emits plain text. On GitHub Actions Windows runners, the auto-detection thinks colors are supported even with captured stdout.

## Fix

`.github/workflows/test.yml`:
```yaml
      - name: Run pytest
        env:
          QT_QPA_PLATFORM: offscreen
          PYTHONIOENCODING: utf-8
          NO_COLOR: "1"        # v1.7.45 NEW
        run: |
          pytest tests/ -q --tb=line --timeout=120
```

[NO_COLOR](https://no-color.org) is the official convention — Rich, Click, and Typer all respect it. One env var addresses all 3 Class A failures.

## Authoritative-principle catches

### Catch — environment-driven test flakiness needs environment-driven fixes

The 3 Class A tests aren't wrong — they verify option names are documented in help output. They just assumed plain-text rendering, true everywhere except this particular runner. Making the tests ANSI-aware (strip-then-assert) would be more code than the env var and brittler — Rich's ANSI patterns could change between versions.

### Design discipline — one-class-per-ship

Each fix in its own commit + release notes so the trail is clear about what changed for what reason.

No new lesson — lesson #61 in concrete form, occurrence #3.

## Limitations

- **Recycle-bin failure remains in CI.** Will be addressed in v1.7.46.
- **`NO_COLOR=1` affects all of CI.** Intentional — any color-sensitive test would also be affected. Verified locally that no tests break with the env var set.
- **Local dev unchanged.** Developers don't need to set NO_COLOR; Rich's local auto-detection already does the right thing.
