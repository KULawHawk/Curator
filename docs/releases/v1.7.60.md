# v1.7.60 — PEP 508 marker on pywin32 (CI green attempt 2)

**Date:** 2026-05-12
**Type:** CI infrastructure / cross-platform install fix
**Scope:** `pyproject.toml` (+marker)

## TL;DR

v1.7.59 fixed the test-collection crash on Linux/macOS but exposed a deeper failure layer: **pip install crashed because `[windows]` extras contain pywin32, which has no macOS/Linux wheel.** Fixed with PEP 508 environment marker `; sys_platform == 'win32'` on the pywin32 line.

## The discovery

v1.7.59 pushed; CI run started on a6be12e. Within 60s, all 3 macOS cells reported failure. Same surface annotation as before ("No files were found with the provided path: coverage.xml"), but step-level inspection revealed the failure had moved: **step 5 (Install Curator) now failing**, where previously step 6 (Run pytest) failed.

That's progress. The pipeline is reaching further before crashing.

## Root cause

```toml
windows = [
    "pywin32>=306",         # <-- Windows-only, no macOS wheel
    "pystray>=0.19",        # cross-platform
    "APScheduler>=3.10",    # cross-platform
]

all = ["curator[dev,beta,cloud,organize,windows,gui,mcp]"]
                                          ^^^^^^^ pulls pywin32 everywhere
```

On macOS CI, `pip install -e ".[all]"` resolves to `[windows]`, tries to install pywin32, fails. Pip aborts before pytest runs. The previous test-collection fix doesn't matter — pytest never gets a chance.

## The fix

```toml
windows = [
    "pywin32>=306 ; sys_platform == 'win32'",  # <-- only on Windows
    "pystray>=0.19",
    "APScheduler>=3.10",
]
```

PEP 508 environment markers are evaluated by pip at resolution time. On Windows: True → pywin32 installs. On macOS/Linux: False → pywin32 silently skipped.
