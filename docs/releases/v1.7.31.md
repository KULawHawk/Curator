# v1.7.31 — `curator audit-export` (append-only audit archival)

**Date:** 2026-05-11
**Type:** New CLI feature
**Scope:** `cli/main.py` (+208 lines, no existing code modified)

## TL;DR

New top-level command `curator audit-export` archives audit log entries to JSONL or CSV files while honoring AuditRepository's append-only design. The DB is never mutated; the file is the durable record. Every export itself emits a `audit.exported` meta-audit so the archival action is part of the trail.

```bash
# Archive everything older than 90 days
curator audit-export --to old.jsonl --older-than 90

# Export Q1 2026 migration events as CSV for spreadsheet review
curator audit-export \
  --to migrations_q1.csv \
  --since 2026-01-01 --before 2026-04-01 \
  --action migration.move \
  --format csv

# Stream filtered output to stdout for piping
curator audit-export --to - --actor cli.scan | jq '.action' | sort -u
```

## Why this matters

Across 30 ships in the v1.7.x arc the audit log has accumulated continuously. A forensic deployment (e.g., a clinical-grade Curator run capturing every migration / scan / PII match) can rack up tens of thousands of audit rows over a quarter. AuditRepository's docstring is explicit:

> Append-only by design. Forensic immutability is the property. No `delete()` or `prune()` methods exist; events are durable.

That design ruled out an "audit-prune" command earlier in the arc. v1.7.31 takes the complementary path: **archive without delete**. The exported file is the durable record (suitable for cold storage, off-site backup, or compliance review); the live DB stays intact until an operator deliberately decides to spin up a fresh deployment.

`audit-export` is the third member of the audit-CLI family:

| Command | What it does | Mutation |
|---|---|---|
| `curator audit` | Live query, tabular display | none |
| `curator audit-summary` | Aggregate per-actor / per-action / time histogram | none |
| `curator audit-export` (NEW) | Serialize entries to file (JSONL/CSV) | none |

None of these mutate; together they cover query / aggregate / archive.

## Command surface

```
curator audit-export
    --to PATH | -o PATH       Required. Output file; '-' for stdout
    --older-than N            Convenience for --before=now()-Ndays
    --before ISO              Upper time bound (occurred_at < this)
    --since ISO               Lower time bound (occurred_at >= this)
    --actor TEXT              Filter by actor (exact)
    --action TEXT             Filter by action (exact, e.g. 'migration.move')
    --entity-type TEXT        Filter by entity_type (exact, e.g. 'file')
    --format jsonl|csv        Output format; default jsonl
    --limit N                 Safety cap on rows; default 1,000,000
```

Mutual-exclusion rules:
- `--older-than` and `--before` are both upper-bound specifiers; using both is rejected
- `--since` must be earlier than `--before` (basic sanity)

Defensive checks:
- Output path with `.db` extension is refused (prevents footgun of clobbering a SQLite file)
- Negative `--older-than` is rejected
- Invalid ISO datetimes in `--before` or `--since` produce helpful errors

## Output formats

### JSONL (default)

One JSON object per line. Complete record shape preserved:

```jsonl
{"audit_id": 137, "occurred_at": "2026-05-11T19:23:45.123", "actor": "cli.scan", "action": "scan.completed", "entity_type": "source", "entity_id": "local", "details": {"files_seen": 1234, "duration_seconds": 12.3}}
{"audit_id": 138, "occurred_at": "2026-05-11T19:23:51.456", "actor": "cli.migrate", "action": "migration.move", "entity_type": "file", "entity_id": "9c3...", "details": {"src": "/old", "dst": "/new", "size": 8192}}
```

Suitable for:
- Programmatic consumers (jq, custom scripts)
- Re-import into another Curator instance (each line is a complete record)
- Streaming to log-aggregation systems

### CSV

Seven-column header + rows. `details` is JSON-encoded into a single column so the schema stays rectangular:

```csv
audit_id,occurred_at,actor,action,entity_type,entity_id,details_json
137,2026-05-11T19:23:45.123,cli.scan,scan.completed,source,local,"{""files_seen"": 1234, ""duration_seconds"": 12.3}"
138,2026-05-11T19:23:51.456,cli.migrate,migration.move,file,9c3...,"{""src"": ""/old"", ""dst"": ""/new"", ""size"": 8192}"
```

Suitable for:
- Spreadsheet review (Excel, LibreOffice, Google Sheets)
- Pivot tables / charts in BI tools
- Auditor-facing summaries where each row corresponds to one event

The `details_json` column is itself a valid JSON value — analysts comfortable with `JSON_VALUE()` / Python `json.loads()` can drill into nested fields per row.

## Meta-audit recording

Every successful export emits a `audit.exported` event:

```json
{
  "actor": "cli.audit",
  "action": "audit.exported",
  "entity_type": "audit_log",
  "entity_id": null,
  "details": {
    "output_path": "old.jsonl",
    "format": "jsonl",
    "rows_exported": 14523,
    "filters": {
      "since": null,
      "before": "2026-02-10T00:00:00",
      "older_than_days": 90,
      "actor": null,
      "action": null,
      "entity_type": null,
      "limit": 1000000
    }
  }
}
```

This means the audit trail records its own archive operations. Years later, the question "when was the first half of 2026 archived, and to where?" is answerable by querying for `action=audit.exported`.

## Design principles reinforced

1. **Append-only stays append-only.** No `delete()` method was added. No `--clear-after-export` shortcut. Anyone wanting to clear the DB after archival must do it as a separate explicit action. Bundling destructive operations with convenience ones is how forensic integrity gets accidentally compromised; keeping them separate keeps the trail trustworthy.

2. **The export records itself.** The `audit.exported` meta-audit captures filters + row count. Operations that affect the audit trail (even just reading it) are themselves auditable.

3. **Defensive output validation.** Refusing `.db` extensions prevents `curator audit-export --to curator.db` from truncating the actual SQLite file. One line of code; prevents unrecoverable data loss.

4. **Use lesson #50 helpers from the start.** New code imports `CHECK` from `curator.cli.util`. No literal `\u2713` glyph. v1.7.30's helper made this the natural default.

## Limitations

- **No streaming for huge exports.** All matching rows are loaded into memory before writing. A 100k-row export needs proportional RAM. Mitigation: use `--limit` to chunk; the existing default of 1M rows is conservative.

- **No compression option.** `.jsonl` and `.csv` are uncompressed. Users wanting `.jsonl.gz` need to pipe through `gzip` separately. Future flag: `--compress gzip|bz2|xz`.

- **No incremental archive bookkeeping.** Each export is independent; the command doesn't remember "last archive was up to audit_id N." Mitigation: use `--since` with the previous run's `--before` value, or add `--after-id N` in a future ship.

- **No GUI exposure.** CLI-only. The existing Audit tab in the GUI could surface an "Export…" button as a future ship.

- **No batch deletion path** (deliberate). Anyone needing to actually shrink the live DB after archival must do it manually with explicit intent — e.g., by spinning up a fresh deployment or running a one-off SQL prune. The complementary delete operation is deliberately NOT bundled.

## What this enables

- Long-term forensic deployments can run continuously without unbounded DB growth becoming a blocker
- Quarterly / monthly archive workflows are now scriptable (`crontab`, scheduled tasks)
- Auditors can be handed CSV files of specific event types without needing live DB access
- The Curator MCP server's audit endpoint (when built) can use the same JSONL format for paginated responses
