# v1.7.30 — Lesson #50 codified: `cli/util.py` ASCII-fallback helper

**Date:** 2026-05-11
**Type:** Refactor / structural defense
**Scope:** `cli/util.py` (new module), `cli/main.py` (audit replacing 8 glyph occurrences), `tests/unit/test_cli_util.py` (new 23-test suite)

## TL;DR

The recurring Unicode-in-CLI-strings bug (lesson #50) hit FIVE times across the v1.7.21 → v1.7.29 arc, most recently as a pre-existing bug in `sources_config` that had been latent in the codebase for an unknown duration. v1.7.30 extracts a reusable helper module (`curator.cli.util`) with 8 glyph constants that automatically substitute ASCII fallbacks under non-TTY conditions, and replaces all 8 remaining literal Unicode glyphs in `cli/main.py`. The crash mode is now closed by structure rather than vigilance.

## The new module: `curator.cli.util`

133 lines. Three exports plus 8 module-level constants.

### Detection: `_stdout_supports_unicode() -> bool`

Cached (`@lru_cache(maxsize=1)`) because the answer is process-lifetime-stable. Truth table:

```
isatty=False (subprocess pipe, file redirect)  -> False
isatty=True  + encoding='utf-8' / 'UTF-8' / 'utf8' / 'utf-16'  -> True
isatty=True  + encoding='cp1252' / 'latin-1' / 'ascii'         -> False
isatty=True  + encoding=None (rare edge case)                  -> False
```

The detection is straightforward — only mark "supports Unicode" when stdout is interactive AND its encoding starts with `utf`. Anything else gets ASCII fallbacks.

### Fallback table: `_GLYPH_FALLBACKS`

Dispatch table (the same pattern used in v1.7.28's PII enrichment and v1.7.29's source visibility):

```python
_GLYPH_FALLBACKS: dict[str, str] = {
    "\u2588": "#",     # FULL BLOCK -- audit-summary histograms (v1.7.21)
    "\u2713": "[OK]",  # CHECK MARK -- success lines (v1.7.28, v1.7.29)
    "\u2717": "[X]",   # BALLOT X -- error/failure lines (v1.7.30)
    "\u2192": "->",    # RIGHTWARDS ARROW -- migration/tier "src -> dst" (v1.7.25)
    "\u2190": "<-",    # LEFTWARDS ARROW -- lineage display "a <- b" (v1.7.30)
    "\u2026": "...",   # HORIZONTAL ELLIPSIS -- "and N more" tails (v1.7.25)
    "\u00d7": "x",     # MULTIPLICATION SIGN -- dimensions
    "\u26a0": "!",     # WARNING SIGN -- inline warnings (v1.7.30)
}
```

Adding a new entry is a one-line table addition. Removing entries is risky (would silently break callers using that constant) — additions only.

### Convenience constants

```python
CHECK = _const("\u2713")      # "✓" or "[OK]"
CROSS = _const("\u2717")      # "✗" or "[X]"
ARROW = _const("\u2192")      # "→" or "->"
LARROW = _const("\u2190")     # "←" or "<-"
ELLIPSIS = _const("\u2026")   # "…" or "..."
BLOCK = _const("\u2588")      # "█" or "#"
TIMES = _const("\u00d7")      # "×" or "x"
WARN = _const("\u26a0")       # "⚠" or "!"
```

Computed at import time (the underlying `_stdout_supports_unicode()` is cached). Each constant resolves to the Unicode glyph in a TTY+UTF-8 environment and the ASCII fallback elsewhere.

### Ad-hoc substitution: `safe_glyphs(text: str) -> str`

For arbitrary text (e.g. from DB rows, user input, search results) that might contain dangerous glyphs:

```python
from curator.cli.util import safe_glyphs

# user_input came from a non-trusted source
console.print(safe_glyphs(user_input))
```

Under TTY+UTF-8, returns input unchanged. Otherwise, replaces every entry of `_GLYPH_FALLBACKS` with its ASCII fallback. Unknown glyphs pass through (they haven't crashed cp1252 yet; adding them is a one-line `_GLYPH_FALLBACKS` update when they do).

## Workflow example

Before v1.7.30 — typical code shape:

```python
console.print(f"[green]\u2713[/] Migration complete.")
console.print(f"  {src} \u2192 {dst}")
console.print(f"  {n} files migrated\u2026")
```

Each line is a latent cp1252 crash. Subprocess tests would crash if they exercised this code.

After v1.7.30:

```python
from curator.cli.util import CHECK, ARROW, ELLIPSIS

console.print(f"[green]{CHECK}[/] Migration complete.")
console.print(f"  {src} {ARROW} {dst}")
console.print(f"  {n} files migrated{ELLIPSIS}")
```

In a Windows Terminal session: identical output to before. In a subprocess capture: clean ASCII output, no crash. The code reads as naturally; the crash surface is gone.

## Limitations

- **Module discovery is import-based.** A future contributor who doesn't know about `cli/util.py` and writes literal `\u2713` is still a latent crash bug. The cure is one of:
  - A CI lint scanning `src/` for codepoints in `_GLYPH_FALLBACKS`
  - A pre-commit hook with the same scan
  - Code review discipline (least reliable)

- **Fallback table is curated, not exhaustive.** Emojis, mathematical symbols, CJK glyphs, etc. would still crash if introduced. The 8-entry table is the empirical set from the arc; additions are one-liners.

- **GUI code unaudited.** `gui/dialogs.py` and `gui/models.py` contain ~13 Unicode glyphs in Qt widget strings. These are safe (Qt uses UTF-16 internally and never hits the cp1252 encoder). Flagging for future audit if any Qt text ever flows to a CLI `print()` test capture.

- **The `TIMES` constant is unused** in main.py — no `×` glyph found in current `cli/main.py`. Included for completeness because dimensions/multiplication (`300×300`, `8×8`) are common in future output; pre-stocking the fallback is cheaper than re-discovering the bug.

## What this closes

- Lesson #50 (the 5-strike recurrence)
- The implicit dependency on contributor vigilance for cp1252 safety
- The latent-bug surface from any future addition of `\u2713`, `→`, `←`, `…`, `█`, `×`, `⚠`, `✗` in cli/main.py without thought

## What this enables

- Subprocess-based testing of any CLI path can now be done without fear of encoding crashes
- New CLI commands can use natural-looking glyphs (`f"[green]{CHECK}[/]"` instead of awkward conditional ASCII)
- The dispatch-table pattern (now used 3 times this arc) is a proven template for future similar problems
