# v1.7.40 — GUI source-edit dialog (closes v1.7.39 limitation 1)

**Date:** 2026-05-11
**Type:** GUI parity (closing the loop on v1.7.39)
**Scope:** `src/curator/gui/dialogs.py` (+130), `src/curator/gui/main_window.py` (+55), 1 new test file (+280, 15 tests)

## TL;DR

Right-click a row in the Sources tab → **Properties...** opens `SourceAddDialog` in edit mode. Users can now change `share_visibility`, `display_name`, `enabled`, and any plugin config field on an existing source via the GUI — no more CLI fallback. The dialog handles both create + update paths via a single optional `editing_source` constructor parameter (DRY); the new branches are small, clearly marked, and don't disturb the v1.7.39 add path.

## Why this matters

v1.7.39 shipped `share_visibility` as a dropdown in `SourceAddDialog`, but only on **creation**. Changing an EXISTING source's visibility still required:

```
curator sources config <id> --share-visibility public
```

v1.7.39's release notes explicitly listed this as the top remaining GUI gap. v1.7.40 closes it.

### Before / after

```
# Before v1.7.40:
GUI:  [right-click source row]  ->  Enable | Disable | Remove
CLI:  curator sources config my_archive --share-visibility public

# After v1.7.40:
GUI:  [right-click source row]  ->  Properties... | (separator)
                                    Enable | Disable | (separator)
                                    Remove
      Properties... opens SourceAddDialog in edit mode,
      pre-filled with the source's current state.
```

## What's new

### `SourceAddDialog` edit mode (`src/curator/gui/dialogs.py`, +130 lines)

**New `__init__` parameter**:
```python
def __init__(
    self,
    runtime: "CuratorRuntime",
    parent: QWidget | None = None,
    *,
    editing_source: "SourceConfig | None" = None,  # v1.7.40
) -> None:
```

**Three new properties**:
- `is_edit_mode` — True if constructed with `editing_source`
- `saved_source_id` — semantic alias for `created_source_id` (latter is misleading for updates; kept for back-compat)
- `created_source_id` — unchanged (returns the source_id that was inserted OR updated)

**New `_prefill_from_source(src)` helper** runs in `__init__` AFTER `_on_source_type_changed`. It:
1. Sets source_id text + makes the QLineEdit read-only with an immutability tooltip
2. Sets source_type combobox + disables it (changing source_type would invalidate the config_json against a different schema)
3. Calls `_on_source_type_changed()` again to rebuild plugin config widgets for the (now-locked) source_type
4. Fills each plugin config widget from `src.config[field_name]`:
   - `QPlainTextEdit` (array fields) ← list joined with newlines
   - `QCheckBox` ← bool
   - `QLineEdit` ← str
5. Fills `display_name`, `enabled`, `share_visibility`

**Window title** switches from `Curator - Add source` to `Curator - Edit source: <source_id>` in edit mode.

**`_on_ok_clicked` mode switch**:
```python
if self._is_edit_mode and self._editing_source is not None:
    created_at = self._editing_source.created_at or datetime.now()
else:
    created_at = datetime.now()

src = SourceConfig(... created_at=created_at, ...)
if self._is_edit_mode:
    self.runtime.source_repo.update(src)
else:
    self.runtime.source_repo.insert(src)
```

The `created_at` preservation is critical: edits do NOT reset the timestamp to now, so sorting/filtering by creation date stays accurate.

Properties...           ← NEW

**New slot** `_slot_source_edit_properties(source_id)`:
```python
def _slot_source_edit_properties(self, source_id: str) -> None:
    try:
        existing = self.runtime.source_repo.get(source_id)
    except Exception as e:
        QMessageBox.critical(...)
        return
    if existing is None:
        # Source removed between right-click and click -- show warning,
        # refresh table.
        QMessageBox.warning(...)
        self._refresh_sources_table()
        return

    dlg = SourceAddDialog(self.runtime, self, editing_source=existing)
    if dlg.exec() == QDialog.DialogCode.Accepted and dlg.saved_source_id:
        self._refresh_sources_table()
        QMessageBox.information(...)
```

All failure paths use descriptive QMessageBox dialogs (no silent log writes).

## Authoritative-principle catches

### Catch — `_prefill_from_source` must run AFTER `_on_source_type_changed`

The dialog flow has a subtle ordering: `__init__` calls `_on_source_type_changed()` first (which builds plugin config widgets for the alphabetically-first plugin), THEN in edit mode `_prefill_from_source` runs which:
1. Sets source_type to the actual stored value
2. Calls `_on_source_type_changed()` again to rebuild widgets for the right plugin
3. Then fills the widgets

The double `_on_source_type_changed` call is cheap. Doing this in the wrong order would either build widgets for the wrong plugin OR try to fill widgets that don't exist yet.

### Design note — source_type is mutable in SQL but immutable in GUI

`SourceRepository.update()` will happily UPDATE the source_type column if you pass a different value. But doing so would invalidate the existing `config_json` (validated against a different plugin's schema). The GUI enforces immutability via a disabled combobox + tooltip. The CLI's `curator sources config` doesn't expose a `--source-type` flag either, so this matches the established surface area.

A future hardening could add repository-layer validation that rejects source_type changes outright — but that's a separate ship.

### No new lesson codified

The patterns here (create-or-update form, prefill-after-build sequencing, mode flags) are well-established UI patterns. Lessons #46–#59 unchanged.

## Limitations

- **No "Cancel changes" preview.** Once OK is clicked, the update is final. Users wanting to compare before/after can use `git log` on the DB or `curator audit` (v1.7.31 audit-export records source-config mutations).
- **No bulk edit.** Right-click only shows the menu for the first selection. Bulk "make these 5 sources public" is a future ship.
- **source_type immutability is GUI-only.** Repository-level validation that rejects source_type changes is queued.
- **No config-field schema validation on re-save.** If a user manually edited the DB and broke a config field, the dialog will prefill with the broken value and re-save it. `_collect_config` catches most issues but not all.
