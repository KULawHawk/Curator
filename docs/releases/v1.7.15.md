## What's new

| Name | Severity | Regex |
|---|---|---|
| `jwt` | HIGH | `\beyJ[A-Za-z0-9_\-]{10,}\.eyJ[A-Za-z0-9_\-]{10,}\.[A-Za-z0-9_\-]{20,}\b` |
| `gitlab_pat` | HIGH | `\bglpat-[A-Za-z0-9_\-]{20,}\b` |
| `atlassian_api_token` | HIGH | `\bATATT3xFfGF0[A-Za-z0-9_\-=]{20,}\b` |

### The dual-eyJ insight

JWT base64-encodes a JSON header `{"alg":...}` and payload `{"sub":...}`. Both objects start with `{"`. The base64 encoding of `{"` is **always** `eyJ`. So every legitimate JWT has the form `eyJ<...>.eyJ<...>.<sig>`.

Requiring **both** segments to start with `eyJ` cleanly rejects:
- Discord bot tokens (`[MN]<...>.<...>.<...>` — wrong prefix on segment 1)
- Random 3-segment dotted strings (very unlikely to start with `eyJ` twice by coincidence)
- Test fixtures with `xxx.yyy.zzz` patterns (no `eyJ` prefix at all)

Test 4 of the new suite explicitly verifies that JWT and Discord don't double-match the same token.

## Authoritative-principle catches

**0 bugs caught.** Clean first-try ship. Lesson #45 (programmatic test data construction) applied from the start — every length-sensitive test input was built with `"a" * N` and asserted before the regex test, so no manual counting errors.

The dual-eyJ design worked first try. JWT and Discord both have 3-segment dot formats but their prefix constraints (`eyJ` vs `[MN]`) keep them mutually exclusive without any additional logic.

## Limitations / next steps

- **No JWT signature validation** — out of scope (would need the secret/public key)
- **No JWT payload parsing** — could base64-decode for audit-log enrichment in v1.8
- **No GitLab `glcbt-` / `glcdt-` variants** — less common; could add if demand surfaces
- **No Bitbucket app password** — no distinctive prefix
- **No Linear/Notion/Vercel/Sentry tokens** — could batch-add later
- **No Conclave hookspec for custom validators** — still on v1.8 list
