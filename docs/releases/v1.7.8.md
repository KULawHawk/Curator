# Curator v1.7.8 — Release Notes

**Tagged:** 2026-05-11

---

## Headline

New `curator tier <recipe>` command identifies files that have aged into a different storage tier. Three named recipes — **cold** (stale provisional), **expired** (past expires_at), **archive** (stale vital) — surface migration candidates based on the T-C02 status taxonomy. Detect-only baseline; emits `tier.suggest` audit events.

---

## What's new

### Service layer: `services/tier.py` (~280 lines)

```python
from curator.services.tier import TierService, TierCriteria, TierRecipe

svc = TierService(file_repo)
report = svc.scan(TierCriteria(
    recipe=TierRecipe.COLD,
    min_age_days=90,
    source_id="local",      # optional filter
    root_prefix="C:/Work",  # optional filter
))

print(report.candidate_count)   # 47
print(report.total_size)        # 12_345_678 bytes
print(report.by_source())       # {"local": 47}
for c in report.candidates:
    print(c.file.source_path, "-", c.reason)
```

### 3 named recipes

| Recipe | Selects | Default age |
|---|---|---:|
| **cold** | `status='provisional'` AND `last_scanned_at` older than threshold | 90 days |
| **expired** | `expires_at IS NOT NULL` AND `expires_at < now` | n/a |
| **archive** | `status='vital'` AND `last_scanned_at` older than threshold | 365 days |

### CLI: `curator tier <recipe>`

```
$ curator tier cold --show-files

Tier scan: cold
  Recipe:          cold
  Min age:         90 days
  Duration:        0.024s
  Candidates:      3
  Total size:      4.2 MB
  By source:
    local: 3

Candidates (oldest-staleness first):
  C:/Users/jmlee/Downloads/old_draft.docx
      512.0 KB  -  provisional, last scanned 300d ago
  ...
```

Flags:
- `--min-age-days N` (override per-recipe default; ignored for `expired`)
- `--source-id X` (restrict to one source)
- `--root PREFIX` (restrict to source_path prefix)
- `--show-files` (per-candidate paths + reasons; default = summary only)
- `--limit M` (cap displayed candidates after oldest-first sort)
- `--json` (machine-readable output)

## Limitations / next steps

- **No one-step `--apply --target <dst>`** — detect-only. The natural follow-up is to chain into `MigrationService.create_job()`. Deferred to v1.8 because it requires a `--target-source-id` design pass (cold storage could be local disk, gdrive, B2, etc.).
- **No daemon mode** — the user runs `curator tier` when they want to. A v1.8 watchdog could schedule it and queue results in an Inbox tab.
- **No interactive picker** — candidates are listed but not selectable. A future GUI tab could show a checklist with bulk-actions ("Send these 47 cold candidates to gdrive:archive").
- **No tier-by-size** — e.g. "files >1 GB regardless of status." Could add as a 4th recipe (`bulky`) if size-based lifecycle becomes common.
- **No tier-by-access-pattern** — we use `last_scanned_at` as a proxy for staleness (the time Curator last *saw* the file, not the time anyone *opened* it). A v1.8 watchdog (T-A03) tracking open events would enable true access-based tiering.
- **No reverse-tier (warmup)** — promoting a file from archive back to active is purely a manual `curator status set` operation.
