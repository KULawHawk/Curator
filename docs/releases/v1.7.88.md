## Lesson captured

**Lesson #88 — Multi-ship arcs need explicit scope plans. The plan IS a ship.**

Lessons #76 and #82 both pointed at this. Until now, the doctrine had no protocol for a module too big for one ship. The answer: when a module's effort estimate exceeds ~2x a typical Phase Gamma ship, **write the scope plan first, ship it as the arc's opening move, and only then execute the sub-ships.**

The plan ship deliverables:
- Inventory of uncovered code grouped by logical seam
- Mapping of clusters to sub-ship boundaries with effort estimates
- Stub-infrastructure pre-requirements per sub-ship
- Arc protocol (clean closure per sub-ship, no mid-arc compromise)
- Status tracker updated per ship

Why the plan deserves its own ship rather than a section in v1.7.89's CHANGELOG: **the planning is the actual deliverable.** If v1.7.89 slips for any reason (different priorities, a different session), the plan still exists at a known commit and tag. The arc has a documented contract from the start, not bolted on retrospectively. This is the inverse of "mid-ship is unacceptable" — mid-arc is also unacceptable, and the plan ship is what prevents it.

The corollary: when a multi-ship arc completes, its final ship can amend the arc plan's status tracker AND add an audit-check test to hold the standard going forward.
