# Curator v1.7.26 — Release Notes

**Tagged:** 2026-05-11
**Tag:** `v1.7.26`
**Theme:** JWT payload parsing — PII scanner enrichment for forensic triage.

---

## Headline

When the PII scanner detects a JWT (pattern shipped in v1.7.15), it now decodes the header + payload and surfaces the most useful claims — `alg`, `iss`, `sub`, `aud`, `exp`, `iat`, `kid`, plus derived `exp_iso` and `expired` — as a `metadata` dict on the match. Stdlib-only; **never verifies signatures** (out of scope for a scanner).

---

## Why this matters

v1.7.15 detected JWTs but left analysts with just `pattern=jwt redacted=*****hars`. To answer the obvious questions ("Symmetric or asymmetric? Issuer? Still hot or expired?") they had to paste tokens into jwt.io.

v1.7.26 closes that loop. Triage signals at a glance:

| Signal | Forensic meaning |
|---|---|
| `alg=HS256` | Symmetric signing. If the signing key is in the same scan target, **double exposure**. |
| `alg=RS256` | Asymmetric. Signing key not in scan target. Lower crypto risk; still PII. |
| `expired=true` | Stale credential. Lower severity (can't be reused) but `sub` may still leak identity. |
| `expired=false` + future `exp_iso` | **Active credential.** HIGH triage priority. |
| `iss` | Which auth system to call for revocation. |
| `sub` | Whose access has been exposed. |

---

## What's new

### Scanner extension (`pii_scanner.py`)

- **`_parse_jwt(token) -> dict | None`** — module-level helper. Decodes header (claims: `alg`, `typ`, `kid`), payload (claims: `iss`, `sub`, `aud`, `jti`), and numeric timestamps (`exp`, `iat`, `nbf`). Computes derived `exp_iso` (ISO UTC) and `expired` (bool). Returns `None` on any parse failure.
- **`PIIMatch.metadata: dict | None = None`** — new optional field on the dataclass. JWT matches populated; all others `None` (backward-compatible).
- **`scan_text` enrichment** — single-line addition: `metadata=_parse_jwt(matched) if pat.name == "jwt" else None`.

### CLI emission (`scan-pii`)

- **JSON**: match dicts include `metadata` key.
- **CSV per-match**: new `metadata` column. Encoded as `key=value;key=value` (semicolon-joined, same pattern as v1.7.22's `by_pattern`). Per-file mode unchanged.
- **Rich pretty-print**: when `--show-matches` and pattern is `jwt`, prints a colored sub-line:
  ```
  L  42         jwt  ***hars
             alg=RS256  iss=https://auth.example.com  sub=user@example.com  exp_iso=2030-01-01T00:00:00+00:00  expired=False
  ```
  **Red** when `expired=true`; **dim** otherwise. Eye lands on active credentials first.

## Authoritative-principle this turn

**0 bugs caught** — clean first-try ship across all 10 tests.

**Design lessons reinforced:**

1. **Backward-compatible dataclass extension** — `metadata: dict | None = None` with a default means every existing `PIIMatch(...)` call site keeps working unchanged.

2. **Single-cell CSV encoding** — reused v1.7.22's `name=count;name=count` pattern. Avoided JSON-in-CSV escaping hell. 9-field metadata fits in one cell; parses with `dict(p.split("=", 1) for p in s.split(";"))`.

3. **Stdlib-only** — PyJWT adds 1.5 MB for what's 60 lines of base64 + json + dict-shuffling. Since we never verify signatures (out of scope), PyJWT's main feature is unused anyway.

4. **Color the urgent thing** — red for `expired=true`, dim otherwise. Visual hierarchy matches forensic priority.

---

## Limitations

- **No signature verification** — deliberate. Triage doesn't need crypto validity; revocation does, and that's the auth system's job.
- **No JWKS resolution** — `kid` extracted but not resolved against a JWKS endpoint (would require network access).
- **Only JWT enriched** — other patterns (Stripe live vs test, AWS region) could grow enrichment later.
- **No `--no-metadata` flag** — always on; cost is ~10 μs per JWT match.
- **`metadata` field is generic** — JWT-specific shape. If more patterns grow enrichment, formalize as TypedDict-per-pattern.
