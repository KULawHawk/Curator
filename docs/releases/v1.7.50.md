## Why fail-fast: false

The default `fail-fast: true` cancels other matrix entries on first failure. That hides crucial information — when 3.11 fails, we want to know whether 3.12 and 3.13 also fail (universal regression) or pass (3.11-specific issue). The matrix exists precisely to answer that question; cancel-on-first defeats it.

## Wall-clock impact

GitHub Actions runs matrix entries in parallel, so wall-clock CI latency stays at ~3-5 min. Total runner-minutes triple (~12 → ~36), but GitHub's free tier on public repos has effectively unlimited Linux/Windows minutes.

## Authoritative-principle catches

- **`fail-fast: false` is non-negotiable.** Initial draft used the default. Switching it preserves diagnostic info when one version fails.
- **No local pre-validation possible.** I only have 3.13 installed. The CI run is the first time anyone tests 3.11/3.12. Pushed deliberately; if breakage surfaces, it becomes v1.7.51.
- **3.14 deliberately excluded.** Python 3.14 is in beta; adding it would create false failures from beta-specific issues. When 3.14 ships stable, we'll add it and validate v1.7.47's `utcnow_naive` shim works post-`utcnow()` removal.

## Limitations

- **Only Windows.** OS matrix (C6) is a separate ship.
- **Latest patch only.** GitHub installs the latest 3.11.x; won't catch a 3.11.0-specific regression.
- **No 3.10 or earlier.** pyproject doesn't claim support; we don't test.
- **No 3.14 yet.** Beta; will add post-stable release.
